# Phase 6 — Infrastructure Layer: Task Analysis

> **Purpose of this document:** Assess the necessity, feasibility, and scope of each Phase 6 task given the full body of work completed through Phase 5. For each task, the conclusion states whether it is required, what is already done, and what still needs to be built.

---

## Context: What Phase 5 Left Open

After Phase 5 the system has a complete application layer — use cases, workflows, LangGraph graphs, ports, DTOs, and shell adapters. Every boundary the system needs to communicate across is defined as an abstract interface. None of those interfaces has a concrete implementation yet. That is the job of Phase 6.

The one significant exception is the ingestion pipeline (`src/ingestion/`), which was built earlier and already contains working infrastructure: a ChromaDB persistent vector store writer, an OpenAI embeddings client, SQLite and PDF source readers. The read side of that vector store — the runtime query layer — still needs to be wired.

---

## Task 1 — Adapter Generator

### What the skill covers

Concrete implementations of application ports that connect the system to external technology: LLM APIs, vector stores, databases, file storage.

### What is already done

The two **cross-context adapters** (`DecisionIntelligenceAdapter` and `KnowledgeRetrievalAdapter`) were built in Phase 5 as shell-layer adapters. These translate between bounded contexts. They fulfill the port contracts for `IDecisionIntelligenceService` and `IKnowledgeRetrievalService` and do not need to be rebuilt.

The ingestion pipeline also already implements `IVectorStoreWriter` via `ChromaVectorStoreWriter` — a fully working OpenAI + ChromaDB write adapter.

### What still needs to be built

Five application-layer ports require LLM adapters. All five are intra-bounded-context and call OpenAI.

| Port | Bounded Context | Adapter to build | Technology |
|---|---|---|---|
| `IIntentClassifier` | BC-1 | `OpenAIIntentClassifier` | OpenAI chat completion |
| `IResponseGenerator` | BC-1 | `OpenAIResponseGenerator` | OpenAI chat completion |
| `IResponseEvaluator` | BC-1 | `OpenAIResponseEvaluator` | OpenAI chat completion |
| `ISignalExtractor` | BC-2 | `OpenAISignalExtractor` | OpenAI chat completion |
| `IEvidenceEvaluator` | BC-2 | `OpenAIEvidenceEvaluator` | OpenAI chat completion |

Each adapter follows the same pattern: receive domain or DTO inputs, construct an OpenAI prompt, parse the structured response, return domain or DTO outputs. No domain logic lives here.

### Verdict

**Required. Partially done.** Shell adapters are complete (Phase 5). Five LLM service adapters are Phase 6 work and are the single most important implementation task in this phase — the system cannot classify intent, generate responses, or evaluate evidence without them.

## Task 2 — Repository Generator

### What the skill covers

Concrete implementations of domain repository interfaces that persist and retrieve aggregates from a storage medium.

### What is already done

Every repository interface is already defined in the domain layer. The following exist as abstract contracts with no concrete implementation:

**BC-1 — `conversation_management` domain ports:**

| Interface | Methods |
|---|---|
| `IConversationSessionRepository` | `save`, `find_by_id`, `find_active_by_user_id` |
| `IConversationHistoryRepository` | `append`, `find_by_user_id` |
| `IUserRepository` | `find_by_id` |
| `ISchoolRepository` | `find_by_user_id` |
| `IPurchaseHistoryRepository` | `find_by_user_id` |

**BC-2 — `decision_intelligence` domain ports:**

| Interface | Methods |
|---|---|
| `IDecisionContextRepository` | `save`, `find_by_session_id` |

**BC-3 — `knowledge_retrieval` domain ports:**

| Interface | Methods |
|---|---|
| `IProductRepository` | `search(query_text, top_k)` |
| `IReferralRuleRepository` | `search(query_text, top_k)` |

The ingestion pipeline has already populated a ChromaDB vector store with products and referral rules. **The read side** — i.e., `IProductRepository` and `IReferralRuleRepository` — connects directly to that existing store. The infrastructure for reading is well-defined and the data is already there.

### Recommended implementation strategy for the prototype

The requirement is a working prototype, not a production-grade database system. This guides the choice of implementation:

| Interface | Recommended implementation | Rationale |
|---|---|---|
| `IProductRepository` | `ProductVectorStoreRepository` (ChromaDB + OpenAI embeddings) | Read side of the existing ingestion vector store |
| `IReferralRuleRepository` | `ReferralRuleVectorStoreRepository` (ChromaDB + OpenAI embeddings) | Same vector store, different collection |
| `IDecisionContextRepository` | `InMemoryDecisionContextRepository` | Session-scoped; does not need to survive restarts for a prototype |
| `IConversationSessionRepository` | `InMemoryConversationSessionRepository` | Same rationale |
| `IConversationHistoryRepository` | `InMemoryConversationHistoryRepository` | Same rationale |
| `IUserRepository` | `JsonFileUserRepository` | Load from a fixture JSON file; adequate for prototype |
| `ISchoolRepository` | `JsonFileSchoolRepository` | Same as above |
| `IPurchaseHistoryRepository` | `JsonFilePurchaseHistoryRepository` | Same as above |

The BC-3 repositories are the most technically interesting because they must embed the query text and search the ChromaDB collections — the same embedding model (`text-embedding-3-small`) and client already used by the ingestion pipeline.

### Verdict

**Required entirely. None done.** All eight repository interfaces need concrete implementations. The BC-3 vector store repositories are the most critical because the RAG pipeline cannot retrieve without them, and they connect to infrastructure that already exists (ChromaDB, OpenAI). The BC-1/BC-2 in-memory implementations are trivial for prototype purposes but necessary to integrate the system end to end.

## Task 3 — Event Bus Generator

### What the skill covers

An event bus infrastructure layer: a publish/subscribe mechanism that receives domain events and dispatches them to registered handlers. Useful when contexts communicate via events asynchronously, or when side effects need to be triggered on state transitions.

### What already exists

The domain layer raises events through the `collect_events()` pattern. The following domain events are defined:

**BC-1 — `conversation_management`:**
- `SessionStarted` — emitted when initialization completes
- `TurnProcessed` — emitted when `complete_turn()` is called
- `SessionClosed` — emitted when `close()` is called

**BC-2 — `decision_intelligence`:**
- `ContextUpdated` — emitted when signals are added
- `GapDetected` — emitted when context gaps are identified
- `ContextReady` — emitted when context reaches retrieval readiness

In all use cases, `collect_events()` is called after the domain method returns. Those drained events are now published synchronously through `LoggingEventPublisher` in the shell layer and written to `data/domain_events.log`.

### Why a full external event bus is not applicable

The system's cross-context communication design (defined in `context_communication.md`) explicitly states:

> *"Domain events are intra-component signals only — they do not drive cross-context communication."*

The reason is architectural: the RAG pipeline is **synchronous and sequential**. Evidence must be retrieved before a response is generated. Introducing an asynchronous event bus for cross-context coordination would break the sequential dependency chain and add complexity with no benefit.

There is no requirement in the specification for async workflows, external broker integration (Kafka, RabbitMQ), or event replay.

### What is applicable

A **minimal in-process synchronous publisher** is useful and architecturally correct. It completes the event-raising contract the domain already fulfills: events are raised, they should reach an observability sink. For this prototype, the implementation is `LoggingEventPublisher`, a shell-layer adapter that:

- drains aggregate events at the use-case boundary
- writes structured log entries to `data/domain_events.log`
- does not dispatch across contexts or require an external broker

This is a small addition, not a full infrastructure concern.

### Verdict

**Applicable in minimal form only.** A full external event bus is not needed and would be over-engineering. A lightweight shell-layer `LoggingEventPublisher` is sufficient for completing the architectural loop and enabling observability. This remains a sub-task of the composition root (dependency wiring), not a standalone event-bus subsystem.

## Task 4 — API Entry Point Generator

### What the skill covers

External entry points that receive HTTP requests (or CLI input, events, webhooks), parse them, construct application commands, call use cases, and return responses. These are the system's external boundary.

### What is already done

Nothing. No API layer exists. The application layer is fully built but has no surface through which an external caller can reach it.

### What needs to be built

The deliverable explicitly requires:
> *"API endpoint — multi-turn conversation — LLM integration — context management"*

The three BC-1 use cases map directly to REST endpoints:

| Use case | HTTP method | Path | Request body | Response |
|---|---|---|---|---|
| `StartSessionUseCase` | `POST` | `/sessions` | `{ "user_id": "..." }` | `{ "session_id": "...", "status": "..." }` |
| `ProcessTurnUseCase` | `POST` | `/sessions/{session_id}/turns` | `{ "user_id": "...", "message": "..." }` | `{ "response_text": "...", "intent": "...", "is_fallback": false, ... }` |
| `CloseSessionUseCase` | `DELETE` | `/sessions/{session_id}` | `{ "user_id": "..." }` | `{ "session_id": "...", "status": "closed" }` |

Recommended framework: **FastAPI**. It is the current Python standard for AI/ML APIs, supports async, produces OpenAPI documentation automatically, and is fast to implement.

The entry point layer's responsibilities are strictly:
- Parse and validate the request body
- Build the use case command dataclass
- Call `use_case.execute(command)`
- Map the response object to a JSON-serialisable dict
- Return the appropriate HTTP status code

No domain logic. No direct imports of domain objects in route handlers.

### Error handling at the boundary

Use cases raise typed exceptions (`SessionNotFoundError`, `SessionConflictError`, `InvalidTurnStateError`, etc.). The entry point layer registers exception handlers that map these to correct HTTP status codes (404, 409, 422).

### Verdict

**Required. Nothing done. Highest external visibility.** This task directly produces the deliverable the evaluators will test. It is the entry point through which all Phase 5 work becomes observable. FastAPI routes for three endpoints, plus an exception handler map, is the full scope.

## Summary and Recommended Execution Order

| Task | Status | Verdict | Priority |
|---|---|---|---|
| **Task 1 — Adapter Generator** | Shell adapters done. 5 LLM adapters missing. | **Required — partially done** | Critical path |
| **Task 2 — Repository Generator** | All interfaces exist. 0 implementations exist. | **Required — nothing done** | Critical path |
| **Task 3 — Event Bus Generator** | Events raised but discarded. No bus exists. | **Minimal form only — in-process InMemoryEventBus** | Low priority, optional for prototype |
| **Task 4 — API Entry Point Generator** | Nothing exists. | **Required — nothing done** | Highest external visibility |

### Suggested order of execution

The constraints are: repositories must exist before adapters can be tested; adapters must exist before the API can return real results; the API is the final assembly point.

```
1. Task 2 — Repositories
       BC-3 first (ProductVectorStoreRepository, ReferralRuleVectorStoreRepository)
       then BC-1/BC-2 in-memory implementations

2. Task 1 — LLM Adapters
       IIntentClassifier, IResponseGenerator, IResponseEvaluator
       ISignalExtractor, IEvidenceEvaluator

3. Task 4 — API Entry Points
       FastAPI app, 3 routes, exception handlers

4. Task 3 — InMemoryEventBus (optional)
       Wire as post-step inside the composition root
       Add logging handlers for SessionStarted, TurnProcessed, GapDetected
```

### What this phase produces

At the completion of Phase 6, the system will be fully runnable end to end:

- A user calls `POST /sessions` → gets a `session_id`
- The user calls `POST /sessions/{session_id}/turns` with a message → gets a grounded response backed by ChromaDB product and referral data
- The user calls `DELETE /sessions/{session_id}` → session is closed

All domain invariants, LangGraph orchestration, RAG pipeline, and response quality evaluation will be active on every request.
