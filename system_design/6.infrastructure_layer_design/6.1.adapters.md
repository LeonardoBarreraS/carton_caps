# Phase 6 — Infrastructure Layer: Adapter Generator

> **Phase inputs:** `use_cases.json`, `context_coordination.json`, `workflow_orchestration.json`  
> **Phase output:** This document + `adapters.json`  
> **Companion document:** `repositories.json` / `repositories.md` — formal repository design record for all persistence adapters

---

## Overview

This document defines all infrastructure adapters for the Carton Caps Conversational Assistant. Adapters implement the application and domain port contracts defined in Phase 5. None of them contain domain logic — they translate between the clean domain layer and the external world.

**17 adapters** are defined across three bounded contexts plus the ingestion pipeline.  
**2 cross-context adapters** (shell layer) were completed in Phase 5 and are not rebuilt here.

---

## Technology Choices

| Concern | Technology | Rationale |
|---|---|---|
| Vector store | **Qdrant** (`qdrant-client`) | Purpose-built vector DB, cosine similarity, Python client, payload filtering. Replaces ChromaDB across both ingestion write and runtime read. |
| Embedding model | **OpenAI `text-embedding-3-small`** | Same model for write (ingestion) and read (runtime) — critical for query-document alignment. |
| Relational data | **SQLite** (`sqlite3` stdlib) | Zero setup. Existing `carton_caps_data.sqlite` carries domain fixture data. `rag_metrics.sqlite` carries observability data. |
| Session state | **Python dict (in-memory)** | Session and context state is turn-scoped. Does not need to survive restarts on a prototype. |
| LLM service | **OpenAI Chat Completions** | `gpt-4o-mini` for classification and evaluation (fast, cheap, JSON mode). `gpt-4o` for response generation (quality matters). |

> **OpenAI JSON mode constraint:** When `response_format={"type": "json_object"}` is used, the OpenAI API requires the literal word `json` to appear somewhere in the request messages (system or user). Absence of this word causes an HTTP 400 error. All four JSON-mode adapters (ADAPT-2, ADAPT-4, ADAPT-5, ADAPT-6) satisfy this by ending their system prompt with an explicit instruction that includes the word "JSON".

## New Application Port

Before the adapters, a new port is added to BC-1's application layer:

### `IRAGMetricsLogger`
**Location:** `src/conversation_management/application/ports/i_rag_metrics_logger.py`

An observability port called by `ProcessTurnUseCase` after every TurnGraph execution. It receives a `TurnMetricsDTO` containing the four RAG evaluation scores and turn context, and persists them to `rag_metrics.sqlite`. This keeps observability out of domain logic while making every turn's quality data accessible for analysis.

```python
@dataclass
class TurnMetricsDTO:
    session_id: str
    turn_number: int
    intent: str
    context_state: str
    context_recall_score: float
    context_precision_score: float
    groundedness_score: float
    relevance_score: float
    is_fallback: bool
    clarification_needed: bool
    evidence_chunk_count: int
    timestamp: str  # ISO 8601
```

#### ADAPT-1 — `QdrantVectorStoreWriter`
**Port:** `IVectorStoreWriter`  
**Location:** `src/ingestion/infrastructure/vector_store/qdrant_writer.py`  
**Replaces:** `ChromaVectorStoreWriter`

The write side of the Qdrant vector store, used during offline ingestion. Generates embeddings via OpenAI and upserts them as points into the named Qdrant collection.

**Key behaviors:**
- `write_batch()` — embeds documents in batches of 500 (OpenAI API limit safety), upserts `PointStruct` objects with `{text, metadata}` payload
- `clear_collection()` — deletes and recreates the collection with cosine distance, ensuring idempotent re-ingestion

### LLM Adapters — BC-1 (Conversation Management)

---

#### ADAPT-2 — `OpenAIIntentClassifier`
**Port:** `IIntentClassifier`  
**Location:** `src/conversation_management/infrastructure/llm/openai_intent_classifier.py`  
**Model:** `gpt-4o-mini`, JSON mode

Classifies every user message into one of six `IntentType` values. Called by the `classify_intent` node in TurnGraph.

**Valid intents:**

| Intent | Meaning |
|---|---|
| `product_inquiry` | User asking about grocery products |
| `referral_question` | User asking about the referral/fundraising program |
| `general_question` | General conversational or off-topic message |
| `clarification_response` | User answering a clarification question the assistant asked |
| `ambiguous` | Unclear — triggers a clarification response |
| `out_of_scope` | Messages clearly unrelated to Carton Caps — routes to a scope-redirect; bypasses Decision Intelligence and retrieval entirely |

**Key behaviors:**
- JSON response format: `{"intent": "<value>"}`
- Falls back to `AMBIGUOUS` on unrecognized values
- System prompt includes Carton Caps context and one-shot disambiguation examples for `product_inquiry` vs `referral_question`
- Final instruction line: *"Respond with a JSON object in exactly this format: `{"intent": "<label>"}`"* — satisfies the OpenAI `json_object` word constraint

---

#### ADAPT-3 — `OpenAIResponseGenerator`
**Port:** `IResponseGenerator`  
**Location:** `src/conversation_management/infrastructure/llm/openai_response_generator.py`  
**Model:** `gpt-4o`

Generates the assistant response from retrieved evidence and DecisionContext snapshot. This is the only adapter using `gpt-4o` — response quality is the product. All three public methods personalise their system prompt with the user's name and school name.

**Key behaviors:**
- `generate(message, evidence, context_snapshot)` — injects evidence chunks as a numbered list; instructs the LLM to cite only those chunks; never invent facts. Extracts `user_name` and `school_name` from `context_snapshot` (defaults: `"there"` / `"your school"`) and formats the system prompt template with both values.
- `generate_clarification(message, clarification_question, user_context)` — wraps the clarification question string into a conversational response. Accepts `user_context: dict` with keys `user_name` and `school_name`; formats the clarification system prompt with both values.
- `generate_redirect(message, conversation_history, user_context, redirect_reason, evidence=None)` — unified redirect method. For `"low_quality"` with non-empty evidence chunks: uses the dedicated `_LOW_QUALITY_WITH_EVIDENCE_SYSTEM` template, injects the chunks as a numbered list, strictly forbids deflection phrases ("I don't have information", "I'm not sure", etc.), instructs the LLM to answer from the partial signal, and requires exactly one focused follow-up question. For `"out_of_scope"` and `"general_question"` (or `"low_quality"` with no chunks): uses `_REDIRECT_SYSTEM` with a reason-specific instruction from `_REDIRECT_REASON_INSTRUCTIONS`. Scores hardcoded to `1.0` — quality gate is not applied.
- Evidence source IDs extracted from the evidence dict and stored in `AssistantResponse.evidence_source_ids`
- **Scores start at 0.0** for `generate()` — `groundedness_score` and `relevance_score` are populated by ADAPT-4 in the next pipeline step

**Prompt strategy:**  
Each method uses a personalised system prompt:
- `generate()` template: *"You are a helpful shopping assistant for {user_name} supporting {school_name}'s fundraising program. Only recommend products or referral details that appear in the provided evidence. Never invent product names, prices, or referral rules."*
- `generate_clarification()` template: *"You are a friendly shopping assistant for Carton Caps helping {user_name} support {school_name}'s fundraising program."*
- `generate_redirect()`: splits on `redirect_reason` — for `"out_of_scope"` and `"general_question"`, uses `_REDIRECT_SYSTEM` + `_REDIRECT_REASON_INSTRUCTIONS` (shared template, tone adapts to reason); for `"low_quality"` with available chunks, uses the dedicated `_LOW_QUALITY_WITH_EVIDENCE_SYSTEM` template: injects chunks as a numbered list, explicitly prohibits any form of "I don't have information" deflection, requires the LLM to share what the evidence suggests (even if partial), and mandates one focused follow-up question at the end.

`user_name` and `school_name` are pre-loaded by `ProcessTurnUseCase` from `IUserRepository` and `ISchoolRepository` before TurnGraph invocation, then stored in `TurnGraphState` and forwarded to each node that calls this adapter.

#### ADAPT-4 — `OpenAIResponseEvaluator`
**Port:** `IResponseEvaluator`  
**Location:** `src/conversation_management/infrastructure/llm/openai_response_evaluator.py`  
**Model:** `gpt-4o-mini`, JSON mode

Quality gate for generated responses. Computes two scores that the TurnGraph uses to decide whether to deliver the response or route to a fallback.

**Scores computed:**

| Score | Technique | Meaning |
|---|---|---|
| `groundedness_score` | Claim tracing | Are all claims in the response traceable to the provided evidence chunks? |
| `relevance_score` | Reverse-question | What question does this response answer? Does it match the user's classified intent? |

**Key behaviors:**
- Single OpenAI call with JSON response: `{"groundedness_score": float, "relevance_score": float, "reasoning": str}`
- Scores are clamped to `[0.0, 1.0]`
- Returns a **new** `AssistantResponse` via `dataclasses.replace()` — immutability preserved
- `reasoning` field in response aids debugging (logged to observability or console)
- Final instruction line: *"Return only a valid JSON object."* — satisfies the OpenAI `json_object` word constraint

### LLM Adapters — BC-2 (Decision Intelligence)

#### ADAPT-5 — `OpenAISignalExtractor`
**Port:** `ISignalExtractor`  
**Location:** `src/decision_intelligence/infrastructure/llm/openai_signal_extractor.py`  
**Model:** `gpt-4o-mini`, JSON mode

Extracts structured preference signals from user messages. Called by the `extract_signals` node in DecisionIntelligenceSubgraph. These signals accumulate monotonically in `DecisionContext` (INV-DC-5).

**Key behaviors:**
- JSON response: `{"signals": [{"key": "...", "value": "..."}]}`
- Maps each dict to `PreferenceSignal(key, value, turn_number=turn_number)`
- Returns empty list when no signals detected — never forces a signal extraction
- Extraction is conservative: only what is explicitly stated or clearly implied
- Final instruction line: *"Respond with a JSON object in exactly this format: `{"signals": [...]}`"* — satisfies the OpenAI `json_object` word constraint

**Valid signal keys (exhaustive — only these five keys are emitted):**
- `product_category` — the type of grocery product (cereal, snacks, oatmeal, mac_and_cheese, pasta, rice, vegetables, trail_mix)
- `meal_occasion` — when/how the product is consumed (breakfast, lunch, dinner, snack, school_lunch, on_the_go)
- `dietary_restriction` — explicit dietary constraint ruling out products (gluten_free, nut_free, dairy_free, vegan, halal)
- `health_goal` — nutritional or wellness objective (healthy, whole_grain, high_protein, low_sugar, organic)
- `budget_tier` — price sensitivity (budget, mid_range, premium)

These five keys were selected because their values appear verbatim in the enriched product documents stored in Qdrant (via `ProductSemanticBuilder`). Keys removed from earlier designs — `brand_preference` (no brand data in the 10-product catalog), `family_size` (not encoded in product embeddings), `age_group` (too coarse; overlaps with `meal_occasion=school_lunch`) — were removed because they do not improve vector similarity matching against the actual product corpus.

**`conversation_history`:** The last `_LLM_HISTORY_WINDOW` (5) messages of the conversation history (as OpenAI-format `list[dict]`) are prepended to the messages list. This provides sufficient recent context for accurate signal extraction while bounding per-call token cost.

#### ADAPT-6 — `OpenAIEvidenceEvaluator`
**Port:** `IEvidenceEvaluator`  
**Location:** `src/decision_intelligence/infrastructure/llm/openai_evidence_evaluator.py`  
**Model:** `gpt-4o-mini`, JSON mode

Scores the retrieved evidence before it reaches the response generator. These scores drive the **retry cycle** — if recall falls below threshold, DecisionIntelligenceSubgraph routes back to construct a refined query (INV-RE-3).

**Scores computed:**

| Score | Meaning |
|---|---|
| `context_recall_score` | Does the evidence cover all of the query's information needs? |
| `context_precision_score` | Are the retrieved chunks tightly relevant (vs. noise)? |

**Key behaviors:**
- Presents: (1) retrieval query text, (2) numbered evidence chunks
- JSON response: `{"context_recall_score": float, "context_precision_score": float}`
- Returns `RetrievedEvidence.create()` with same chunks + updated scores — immutability preserved
- Final instruction line: *"Return only a valid JSON object."* — satisfies the OpenAI `json_object` word constraint

#### ADAPT-16 — `SQLiteConversationHistoryRepository`
**Port:** `IConversationHistoryRepository`  
**Location:** `src/conversation_management/infrastructure/repositories/sqlite_conversation_history_repository.py`  
**Technology:** SQLite  
> Repository design: REPO-2 (primary implementation) — see `repositories.md`

Persists `ConversationHistory` records to the existing `Conversation_History` table and reads them back in chronological order. This is the default runtime adapter used by the shell composition root.

**Key behaviors:**
- `append(entry)` — inserts a new conversation history record into SQLite
- `find_by_user_id(user_id)` — reads all messages for a user ordered by timestamp
- Auto-migration on first use adds optional `history_id` and `session_id` columns if missing
- Legacy `sender='bot'` rows are mapped to `SenderType.ASSISTANT` on read (`_SENDER_MAP`)
- On write, domain `SenderType.ASSISTANT` (`'assistant'`) is translated to `'bot'` via `_DB_SENDER_MAP` to satisfy the table's `CHECK(sender IN ('user', 'bot'))` constraint

### Observability Adapter

---

#### ADAPT-15 — `SQLiteRAGMetricsLogger`
**Port:** `IRAGMetricsLogger`  
**Location:** `src/conversation_management/infrastructure/observability/sqlite_rag_metrics_logger.py`  
**DB:** `data/rag_metrics.sqlite` (auto-created)

Every call to `ProcessTurnUseCase` emits one row to the `rag_metrics` table. This is the primary observability surface for the RAG pipeline.

**Table schema:**

| Column | Type | Description |
|---|---|---|
| `id` | INTEGER PK | Auto-incremented row ID |
| `session_id` | TEXT | Conversation session |
| `turn_number` | INTEGER | Turn index within session |
| `intent` | TEXT | Classified intent (product_inquiry, etc.) |
| `context_state` | TEXT | DecisionContext state at retrieval (empty/partial/ready/enriched) |
| `context_recall_score` | REAL | Did vector search find the right evidence? |
| `context_precision_score` | REAL | Were retrieved chunks relevant? |
| `groundedness_score` | REAL | Are response claims traceable to evidence? |
| `relevance_score` | REAL | Does the response answer the question? |
| `is_fallback` | INTEGER | 1 if quality gate failed and fallback was returned |
| `clarification_needed` | INTEGER | 1 if BC-2 detected context gaps |
| `evidence_chunk_count` | INTEGER | Number of chunks retrieved |
| `timestamp` | TEXT | ISO 8601 UTC timestamp |

**Useful queries to understand the RAG pipeline:**

```sql
-- Average scores per session
SELECT session_id, 
       ROUND(AVG(context_recall_score), 2)    AS avg_recall,
       ROUND(AVG(groundedness_score), 2)       AS avg_groundedness,
       ROUND(AVG(relevance_score), 2)          AS avg_relevance
FROM rag_metrics
GROUP BY session_id;

-- Turns where quality gate failed (fallback triggered)
SELECT session_id, turn_number, intent, groundedness_score, relevance_score
FROM rag_metrics
WHERE is_fallback = 1
ORDER BY timestamp DESC;

-- Score breakdown by intent
SELECT intent,
       COUNT(*)                                AS turns,
       ROUND(AVG(context_recall_score), 2)    AS avg_recall,
       ROUND(AVG(groundedness_score), 2)       AS avg_groundedness,
       ROUND(AVG(relevance_score), 2)          AS avg_relevance,
       SUM(is_fallback)                        AS fallbacks
FROM rag_metrics
GROUP BY intent;

-- Turns with low recall (retrieval quality issue — investigate query construction)
SELECT session_id, turn_number, intent, context_state, context_recall_score
FROM rag_metrics
WHERE context_recall_score < 0.6
ORDER BY context_recall_score ASC;
```

To open: `sqlite3 data/rag_metrics.sqlite` or any SQLite viewer (DB Browser for SQLite, DBeaver).

#### ADAPT-17 — `LoggingEventPublisher`
**Port:** `IEventPublisher` (BC-1 and BC-2)  
**Location:** `src/shell/composition/logging_event_publisher.py`  
**Logger:** `carton_caps.domain_events` (Python stdlib `logging`)  
**Default file:** `data/domain_events.log`  
**Config override:** `DOMAIN_EVENT_LOG_PATH`

Publishes domain events drained from aggregates at every use-case boundary so they are no longer fire-and-forget. Placed in the shell layer because it is the only location allowed to import ports from both BC-1 and BC-2 simultaneously. A single instance satisfies both `IEventPublisher` ABCs via Python multiple inheritance and attaches a dedicated file handler for the configured log path.

**Events published:**

| Event | Source aggregate | Emitted when | Use case that publishes |
|---|---|---|---|
| `SessionStarted` | `ConversationSession` | `complete_initialization()` | `StartSessionUseCase` |
| `TurnProcessed` | `ConversationSession` | `complete_turn()` | `ProcessTurnUseCase` |
| `SessionClosed` | `ConversationSession` | `close()` | `CloseSessionUseCase` |
| `ContextUpdated` | `DecisionContext` | `add_signal()` | `ProcessTurnIntelligenceUseCase` |
| `GapDetected` | `DecisionContext` | `record_gap_evaluation()` | `ProcessTurnIntelligenceUseCase` |
| `ContextReady` | `DecisionContext` | `mark_ready()` | `ProcessTurnIntelligenceUseCase` |

**Log sample:**
```
2026-04-12 10:00:00,123 INFO carton_caps.domain_events [DomainEvent] SessionStarted session_id=abc user_id=1 school_id=2 occurred_at=2026-04-12T10:00:00+00:00
2026-04-12 10:00:01,456 INFO carton_caps.domain_events [DomainEvent] ContextUpdated session_id=abc context_state=partial signal_count=1 occurred_at=...

The adapter creates the parent directory for the log file if needed and avoids attaching duplicate file handlers when the container is rebuilt multiple times in the same process.

## Dependency Direction

```
shell/composition        →  BC-1, BC-2, BC-3 application layers
                              (cross-context adapters — Phase 5)

BC-1 infrastructure      →  BC-1 application + domain layers
BC-2 infrastructure      →  BC-2 application + domain layers
BC-3 infrastructure      →  BC-3 domain layer
ingestion infrastructure →  ingestion domain layer

Domain layers            ←  nothing (no infrastructure imports)
```

The dependency arrow always points inward. Infrastructure depends on application and domain. Domain never imports infrastructure.
