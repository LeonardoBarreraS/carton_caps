# Part 1 — For Non-Technical Readers

## What Did We Build in This Phase?

In previous phases we designed the blueprint of the system: what the assistant needs to do, what the rules are, and how the different parts connect. In Phase 4, we turned that blueprint into **real code** — specifically, the core logic of the system.

This core is called the **Domain Layer**. It is the smart center of the application — the part that knows all the rules, enforces them at all times, and never breaks them regardless of what happens around it.

---

## Why Does the Domain Layer Matter?

Think of the Domain Layer as the **referee** of the system.

No matter what the user interface does, no matter what AI model is plugged in, no matter what database is connected — the referee's rules always apply. It cannot be bypassed. If a rule says *"you must initialize a session before using it"*, that check happens inside the code itself, not as a reminder in a comment or a best-practice note. It is enforced automatically, every time, without exception.

This is important because the system handles real fundraising activity for real schools. Delivering invented product recommendations, skipping quality checks, or letting a session enter an illegal state are not performance issues — they are correctness failures that would undermine parent trust. The Domain Layer prevents all of these.

---

## What Are the Three Parts of the System?

The system is organized into three independent areas, each with its own domain logic:

### 1. Conversation Management
This is the "front door" of the assistant. It manages:
- Opening and closing a conversation session
- Tracking what has been said and by whom
- Making sure every turn in the conversation follows the correct sequence
- Storing the final response once quality checks have passed

**Key rule it enforces:** A conversation must always be properly initialized before it becomes active. You cannot skip initialization. You cannot receive a new message while the previous turn is still being processed.

### 2. Decision Intelligence
This is the "brain" of the assistant. It manages:
- Building a growing picture of the user's preferences as the conversation unfolds
- Never forgetting what it has learned (once a signal is recorded, it stays)
- Knowing when it has enough context to make a good recommendation — and when it needs to ask one more question
- Never going "backwards" in its understanding (once it reaches a confident state, it cannot regress)

**Key rule it enforces:** The assistant's context can only grow. It cannot unlearn. And it can never recommend products until it truly understands the user's situation.

### 3. Knowledge Retrieval
This is the "library" of the assistant. It holds:
- The product catalog — all the items available for parents to buy
- The referral program rules — how invitations work, what bonuses apply, what conditions apply

**Key rule it enforces:** Every product or referral rule the assistant mentions must come from this library. The assistant is not allowed to invent, guess, or improvise product names or referral details.

---

## What Makes This Design Safe?

Several key safety guarantees are built directly into the code:

**Every response is grounded.**
If the assistant cannot find real evidence from the product catalog or referral rules, it returns a safe fallback message. It never fabricates a recommendation.

**Every response is scored before delivery.**
Two quality scores are computed after every answer: one for groundedness (is this answer based on evidence?) and one for relevance (does this answer actually address what the user asked?). If either score falls below the threshold, the response is blocked and a safe fallback is returned instead.

**Context never regresses.**
Once the system understands the user's situation, that understanding cannot be lost. Signals accumulate. States only move forward.

**Sessions are strictly controlled.**
A user cannot have two active sessions at once. A closed session cannot be reopened. Every state in a session's lifecycle has exactly one valid path forward.

**The school fundraising goal is always present.**
Every product recommendation explicitly connects back to the user's school. This is not a UI nicety — it is enforced in the code.

---

## In Simple Terms: What Did We Actually Produce?

We wrote **45 Python files** that implement:
- The rules and behaviors of all three parts of the system
- The lifecycle of a conversation session, step by step
- The lifecycle of a user's decision profile, signal by signal
- The product and referral knowledge structures
- The interfaces (contracts) that the AI models, databases, and vector stores must fulfill to connect to the system

None of these files contain AI calls, database calls, or web requests. They contain only **logic and rules**. This keeps the core safe and independent of any technology decision.

---

## Architecture Pattern Applied

**Clean Architecture** — strict dependency direction enforced at all levels:

```
Infrastructure → Application → Domain
```

- Domain layer has no knowledge of application or infrastructure
- All external dependencies are expressed as abstract ports (ABC interfaces)
- Aggregates enforce invariants internally; no service is needed to validate them externally
- Cross-context payloads travel as function return values — never as shared domain object imports

## Python Implementation Conventions

| Concern | Convention |
|---|---|
| All files | `from __future__ import annotations` at top |
| Mutable aggregates | `@dataclass` (regular) |
| Value objects and events | `@dataclass(frozen=True)` |
| Enumerations | `class X(str, Enum)` — string serialization compatibility |
| Ports (interfaces) | `ABC` abstract base classes with `@abstractmethod` |

| Factories | `@classmethod` `create()` or `from_context()` |
| State guards | `_assert_status()` helper on `ConversationSession`; inline guards on `DecisionContext` |

## BC-1 — Conversation Management

### Aggregate Root: `ConversationSession`
**Classification:** Process entity — lifecycle owner of each multi-turn conversation.

**State machine:**
```
created → initializing → idle ⇄ active → closed
```

| Transition | Method | Emits |
|---|---|---|
| `created → initializing` | `begin_initialization()` | — |
| `initializing → idle` | `complete_initialization()` | `SessionStarted` |
| `idle → active` | `receive_message()` | — |
| `active → idle` | `complete_turn(intent, response)` | `TurnProcessed` |
| `idle → closed` | `close()` | `SessionClosed` |

**Illegal transitions** (raise `InvalidSessionTransitionError`):
- `created → active` — initialization cannot be skipped
- `closed → any` — terminal state
- `active → closed` — must idle first

**Factory:** `ConversationSession.create(user_id, school_id) → ConversationSession` — starts in `created` state with `turn_count = 0`.

**Invariants enforced:** INV-CS-1, INV-CS-2, INV-CS-6

---

### Supporting Entities

| Entity | Type | Key Notes |
|---|---|---|
| `ConversationHistory` | `@dataclass(frozen=True)` | Immutable historical entity; append-only log; created once, never mutated |

---

### Value Objects — BC-1

| Value Object | Type | Key Notes |
|---|---|---|
| `User` | `@dataclass(frozen=True)` | External reference; read-only identity reference; never authenticated or mutated here |
| `School` | `@dataclass(frozen=True)` | External reference; read-only identity reference; passed to BC-2 as `SchoolAnchor` |
| `Intent` | `@dataclass(frozen=True)` | Wraps `IntentType` enum; `requires_retrieval()`, `requires_clarification()` methods |

## BC-2 — Decision Intelligence

### Aggregate Root: `DecisionContext`
**Classification:** Process entity — progressively accumulates user preference signals; drives retrieval readiness.

**State machine:**
```
empty → partial ⇄ partial → ready → enriched ⇄ enriched
```

| Transition | Trigger | Emits |
|---|---|---|
| `empty → partial` | `add_signal()` — first non-school signal | `ContextUpdated` |
| `partial → partial` | `add_signal()` — further signals | `ContextUpdated` |
| `partial → ready` | `mark_ready()` — gap evaluation passed | `ContextReady` |
| `ready → enriched` | `add_signal()` after readiness | `ContextUpdated` |
| `enriched → enriched` | `add_signal()` further refinement | `ContextUpdated` |

**Illegal transitions** (raise domain error):
- `ready → partial` or `enriched → any lesser state` — no regression (INV-DC-3)
- `empty → ready` — must pass through `partial`

**Behavior:**
- `add_signal(signal)` — idempotency check prevents duplicates; monotonic accumulation
- `record_gap_evaluation(gaps)` — updates `identified_gaps`; emits `GapDetected` conditionally
- `mark_ready()` — guards: `status == partial` and `len(signals) > 0` (INV-DC-6)
- `is_ready_for_retrieval() → bool` — `True` if `ready` or `enriched`
- `has_gap() → bool` — `True` if `partial` and `identified_gaps` non-empty
- `snapshot() → dict` — read-only primitive snapshot; no mutable internals exposed
- `collect_events()` — drains `_events` list; called by application layer after each operation

**Factory:** `DecisionContext.create(session_id, school_anchor)` — starts `empty`; `school_anchor` cannot be null (INV-DC-2).

---

### Value Objects — BC-2

**`RetrievalQuery`** (`frozen=True`) — Immutable historical artifact value object
- Fields: `query_id`, `session_id`, `query_text`, `source_target`, `originating_intent`, `context_state_at_creation`, `created_at`
- **Type:** Immutable historical artifact; created once via factory, never mutated
- **Factory:** `RetrievalQuery.from_context(session_id, query_text, intent, context_status)`
  - Derives `source_target` from intent via `_INTENT_TO_SOURCE` routing table
  - `product_inquiry → product_catalog`
  - `referral_question → referral_program_rules`
  - Raises `InvalidRetrievalQueryError` on invalid intent mapping (INV-RQ-2, INV-RQ-3)
- **Invariants enforced:** INV-RQ-1, INV-RQ-2, INV-RQ-3

**`SchoolAnchor`** (`frozen=True`) — `school_id`, `school_name`; validates non-empty in `__post_init__`.

**`PreferenceSignal`** (`frozen=True`) — `key`, `value`, `turn_number`; validates `turn_number >= 1`.

**`RetrievedEvidence`** (`frozen=True`) — contains `tuple[EvidenceChunk, ...]`; `context_recall_score` and `context_precision_score` both in `[0, 1]`; `create()` factory validates non-empty chunks.

**`EvidenceChunk`** (`frozen=True`) — `chunk_id`, `source_id`, `content`, `relevance_score`, `groundedness_score`; scores default `0.0` until evaluated.

---

### Domain Events — BC-2

| Event | Emitted By | Key Payload Fields |
|---|---|---|
| `ContextUpdated` | `add_signal()` | `session_id`, `context_state`, `signal_count` |
| `GapDetected` | `record_gap_evaluation()` (conditional) | `session_id`, `gaps: tuple[str, ...]` |
| `ContextReady` | `mark_ready()` | `session_id`, `signal_count` |

`GapDetected` uses a custom `__init__` override to convert `list[str] → tuple[str, ...]` — required for `frozen=True` compatibility.

## BC-3 — Knowledge Retrieval

### Design Note
BC-3 has no aggregate root and no stateful process entities. It is a **read-only knowledge layer**. `Product` and `ReferralRule` are immutable external reference value objects retrieved by semantic similarity at query time — never created or mutated at runtime.

---

### Value Objects — BC-3

**`Product`** (`frozen=True`) — External reference value object
- Fields: `product_id`, `name`, `category`, `brand`, `description`, `attributes: dict[str, str]`
- `__post_init__` validates non-empty `product_id`, `name` only
- **Type:** Immutable external reference; sourced from external knowledge base
- **Invariant enforced:** INV-AR-2 (grounding — only retrieved products reach the LLM) enforced at application layer by `ExecuteRetrievalUseCase`

**`ReferralRule`** (`frozen=True`) — External reference value object
- Fields: `rule_id`, `title`, `description`, `rule_type: RuleType`
- `__post_init__` validates non-empty `rule_id`, `title`, `description`
- `RuleType`: `eligibility`, `bonus`, `invitation`, `requirement`
- **Type:** Immutable external reference; sourced from external knowledge base
- **Invariant enforced:** INV-AR-3 — rules must have non-empty title, description, and valid rule_type

**`RetrievedEvidence`** — local BC-3 definition. Same shape as BC-2 but defined separately — **no shared import across bounded contexts**. Scores default to `0.0`; BC-2's `IEvidenceEvaluator` populates them after retrieval.

---

### Ports — BC-3

| Port | Method Signature |
|---|---|
| `IProductRepository` | `search(query_text: str, top_k: int) → list[Product]` |
| `IReferralRuleRepository` | `search(query_text: str, top_k: int) → list[ReferralRule]` |

## Cross-Context Data Flow

All cross-context payloads travel as function return values — no shared domain object imports between BCs.

```
BC-1 application layer
  │  IDecisionIntelligenceService.process_turn(session_id, intent, message, turn_number)
  ▼
BC-2 application layer
  │  IKnowledgeRetrievalService.retrieve(RetrievalQuery)
  ▼
BC-3 domain layer
  │  IProductRepository.search() or IReferralRuleRepository.search()
  │  returns RetrievedEvidence (raw chunks, scores = 0.0)
  ▲
BC-2 — IEvidenceEvaluator.evaluate() → RetrievedEvidence (scored)
  ▲
BC-1 — IResponseGenerator.generate() → AssistantResponse
       IResponseEvaluator.evaluate() → AssistantResponse (quality-scored)
       ConversationSession.complete_turn(intent, response)
```
