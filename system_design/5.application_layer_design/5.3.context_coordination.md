# Context Coordination — Carton Caps Conversational Assistant

**Phase:** 5 — Application Layer Design  
**Task:** Context Coordination  
**Derived from:** `bounded_contexts.json`, `context_communication.json`, `workflow_orchestration.json`, `use_cases.json`

---

## Design Principles

- Contexts are **autonomous** — no BC domain layer imports from another BC's domain layer.
- Coordination flows through **port interfaces** defined in the calling context's application layer.
- **Adapters in the shell** are the physical cross-context bridges — the only code that imports across component boundaries.
- Every cross-context payload is a **primitive DTO** — no domain object crosses a boundary.
- Coordination logic is **embedded in the workflow graphs** (TurnGraph, DecisionIntelligenceSubgraph) — no additional coordinator class is needed for a synchronous monolith.
- The **shell composition root** is the physical coordination hub — it instantiates adapters and injects them into use cases.
- **Sagas are not needed** — all three BCs are co-located, all calls are synchronous, and each context's write is a single atomic operation at use case boundary.

---

## Coordination Model

**Type:** Synchronous port-based  
**Hub:** `src/shell/composition/`

The RAG pipeline is inherently synchronous: evidence must be retrieved and evaluated before a response can be generated. Async coordination would add complexity without benefit. All three BCs are co-located in the same process.

The **port/adapter pattern** already enforces context autonomy:
- Each BC defines the interface (port) it needs from another context
- The shell wires the adapter that bridges the two BCs
- Each BC sees only its own port interface — never the other BC's internals

The **TurnGraph (WF-GRAPH-1)** is the de facto `ConversationTurnCoordinator`. Its nodes invoke BC-2 and BC-3 through ports. No separate coordinator class is needed.

---

## Cross-Context Workflows

Two cross-context workflows exist in this system:

| Coordinator | Contexts Spanned | Realization | Pattern |
|---|---|---|---|
| `SessionInitializationCoordinator` | BC-1, BC-2 | Embedded in `StartSessionUseCase` (UC-1) | Sequential two-context |
| `ConversationTurnCoordinator` | BC-1, BC-2, BC-3 | Embedded in `TurnGraph` (WF-GRAPH-1, UC-2) | Stateful multi-context pipeline |

## COORD-1: SessionInitializationCoordinator

**Realization:** `StartSessionUseCase` (UC-1)  
**Module:** `src/conversation_management/application/use_cases/start_session_use_case.py`  
**Contexts:** BC-1 → BC-2  
**Trigger:** `StartSessionCommand` from API  
**Pattern:** Sequential — BC-1 leads, BC-2 follows

**Purpose:** Coordinates session creation in BC-1 with `DecisionContext` initialization in BC-2. Session activation is gated on successful seeding of BC-2 — the session cannot become `idle` until a `DecisionContext` anchored to the school exists.

---

### Coordination Flow

```
StartSessionUseCase (BC-1)
  │
  ├── 1. Load User, School, ConversationHistory, PurchaseHistory   [BC-1 repos]
  │
  ├── 2. ConversationSession.create() + begin_initialization()     [BC-1 domain]
  │       session → initializing
  │
  ├── 3. Build PreSeedContextDTO(session_id, school_id, school_name, purchase_signals)
  │
  ├── 4. IDecisionIntelligenceService.pre_seed_context(PreSeedContextDTO)  [BC-1 → BC-2]
  │       └─ DecisionIntelligenceAdapter (shell)
  │            └─ PreSeedContextUseCase (BC-2)
  │                 ├── DecisionContext.create(session_id, school_id, school_name)
  │                 ├── Seed school anchor + purchase signals as initial PreferenceSignals
  │                 └── IDecisionContextRepository.save(context)
  │
  └── 5. session.complete_initialization()                          [BC-1 domain]
          session → idle; emit SessionStarted
          IConversationSessionRepository.save(session)
          return StartSessionResponse
```

**DTO crossing the boundary:**

| DTO | Direction | Fields |
|---|---|---|
| `PreSeedContextDTO` | BC-1 → BC-2 | `session_id`, `school_id`, `school_name`, `purchase_signals: list[str]` |

**Invariants upheld:**
- `INV-CS-3` — school pre-seeded into `DecisionContext` before session becomes active
- `INV-CS-5` — `DecisionContext` seeded at session initialization
- `INV-DC-2` — `DecisionContext` created with school anchor, not blank

## COORD-2: ConversationTurnCoordinator

**Realization:** `TurnGraph` (WF-GRAPH-1) inside `ProcessTurnUseCase` (UC-2)  
**Module:** `src/conversation_management/application/workflows/turn_graph.py`  
**Contexts:** BC-1 → BC-2 → BC-3 → BC-2 → BC-1  
**Trigger:** `ProcessTurnCommand` from API  
**Pattern:** Stateful multi-context pipeline with quality gates and retrieval retry cycle

**Purpose:** Executes the full conversation turn pipeline across all three bounded contexts. Each context performs its autonomous responsibility. The TurnGraph advances the pipeline using conditional edges driven by intent classification, context readiness, and response quality scores.

---

### Coordination Flow

```
ProcessTurnUseCase (BC-1)
  │
  ├── session.receive_message()    [pre-graph: BC-1 domain]
  │
  └── TurnGraph.invoke(TurnGraphState)
        │
        ├── N1: classify_intent                                     [BC-1]
        │       IIntentClassifier.classify(message) → intent: str
        │
        ├── N2: process_decision_intelligence                        [BC-1 → BC-2]
        │       IDecisionIntelligenceService.process_turn(TurnInputDTO)
        │         └─ DecisionIntelligenceAdapter (shell)
        │              └─ ProcessTurnIntelligenceUseCase (BC-2)
        │                   │
        │                   └── DecisionIntelligenceSubgraph.invoke(DecisionGraphState)
        │                         │
        │                         ├── extract_signals               [BC-2]
        │                         │     ISignalExtractor.extract(message, turn_number)
        │                         │
        │                         ├── update_decision_context       [BC-2 domain]
        │                         │     context.add_signal() × N
        │                         │
        │                         ├── evaluate_context_readiness    [BC-2 domain]
        │                         │     ├── intent in _INTENT_SUFFICIENT_RETRIEVAL → construct_retrieval_query (message is query; bypass readiness gate)
        │                         │     ├── gap + not ready → clarification_needed=True → END
        │                         │     ├── general_question / clarification_response → END (no retrieval)
        │                         │     └── ready + retrieval intent → construct_retrieval_query
        │                         │
        │                         ├── construct_retrieval_query     [BC-2 domain]
        │                         │     RetrievalQuery.from_context()
        │                         │     + vague-message backstop: injects prior user message
        │                         │       from conversation_history when assembled content
        │                         │       word count < _MIN_QUERY_CONTENT_WORDS (4)
        │                         │
        │                         ├── execute_retrieval             [BC-2 → BC-3]
        │                         │     IKnowledgeRetrievalService.retrieve(RetrievalQueryDTO)
        │                         │       └─ KnowledgeRetrievalAdapter (shell)
        │                         │            └─ ExecuteRetrievalUseCase (BC-3)
        │                         │                 routes: product_catalog | referral_program_rules
        │                         │                 returns: RetrievedEvidenceDTO → BC-2
        │                         │
        │                         └── evaluate_evidence_quality     [BC-2]
        │                               IEvidenceEvaluator → recall + precision scores
        │                               ├── recall >=  threshold → END (evidence returned to BC-1)
        │                               ├── recall <   threshold AND attempts < MAX → retry
        │                               └── recall <   threshold AND retries exhausted → END (degraded)
        │
        │       ← TurnIntelligenceResultDTO returned to TurnGraph state
        │
        ├── N3: generate_response / generate_clarification          [BC-1]
        │       IResponseGenerator.generate(message, evidence, context_snapshot)
        │
        ├── N4: evaluate_response                                    [BC-1]
        │       IResponseEvaluator.evaluate(response, intent, evidence)
        │       ├── scores pass → persist_turn → END
        │       └── scores fail → return_fallback → END
        │
        [post-graph: BC-1]
        session.complete_turn(intent, response)
        IConversationSessionRepository.save(session)
```

---

### DTOs Crossing Context Boundaries

| DTO | Direction | Fields |
|---|---|---|
| `TurnInputDTO` | BC-1 → BC-2 | `session_id`, `intent: str`, `message: str` |
| `RetrievalQueryDTO` | BC-2 → BC-3 | `query_text: str`, `source_target: str` |
| `RetrievedEvidenceDTO` | BC-3 → BC-2 | `chunks: list[str]`, `evidence_type: str` |
| `TurnIntelligenceResultDTO` | BC-2 → BC-1 | `evidence_chunks`, `evidence_type`, `context_recall_score`, `context_precision_score`, `context_state`, `clarification_needed`, `clarification_question` |

**Rule:** No domain object crosses any boundary. `session_id` (a plain string) is the shared correlation key present in all DTOs.

---

### Conditional Routing Summary

```
classify_intent
  ├── product_inquiry, referral_question, clarification_response   →  process_decision_intelligence
  ├── general_question                                             →  generate_redirect
  ├── out_of_scope                                                 →  handle_out_of_scope
  └── ambiguous                                                    →  generate_clarification

process_decision_intelligence
  ├── clarification_needed == True          →  generate_clarification
  └── clarification_needed == False         →  generate_response

generate_response / generate_clarification  →  evaluate_response   (always)

evaluate_response
  ├── groundedness >= threshold AND relevance >= threshold   →  persist_turn
  └── either score below threshold                           →  return_fallback

persist_turn    →  END
return_fallback →  END
```

### Coordination State

| State carrier | Scope | Lives in |
|---|---|---|
| `TurnGraphState` (16 fields) | Turn — BC-1 orchestration | `WF-GRAPH-1` |
| `DecisionGraphState` (18 fields) | Turn — BC-2 sub-pipeline | `WF-GRAPH-2` |

### Invariants Upheld

| Invariant | Enforced at |
|---|---|
| `INV-WF-1` — intent classified before context update | `classify_intent` node (N1) |
| `INV-WF-2` — context updated before query construction | `update_decision_context` before `construct_retrieval_query` |
| `INV-WF-3` — retrieval complete before response generation | `execute_retrieval` before `generate_response` |
| `INV-WF-4` — Decision Intelligence runs on every turn | `process_decision_intelligence` on all intent paths |
| `INV-DC-4` — retrieval only when context partial/ready/enriched | `evaluate_context_readiness` routing |
| `INV-DC-5` — signals accumulate monotonically | `update_decision_context` — add only, never remove |
| `INV-RQ-1` — query built from context snapshot, not raw message | `construct_retrieval_query` factory |
| `INV-RE-1/2` — evidence quality evaluated before reaching generation | `evaluate_evidence_quality` gate |
| `INV-AR-1, INV-AR-5, INV-AR-6` — response quality gate before delivery | `evaluate_response` → fallback routing |

## Saga Assessment

**Sagas needed: No.**

This is a synchronous co-located monolith. All three bounded contexts run in the same process. There are no distributed transactions, no message queues, and no partial state committed across contexts mid-turn.

**Why rollback is not needed:**
- BC-3 retrieval is **read-only** — no state to undo
- BC-2 `DecisionContext` saved **after** the subgraph completes (post-graph action in UC-5)
- BC-1 `ConversationSession` saved **after** the TurnGraph completes (post-graph action in UC-2)
- If any step fails, the exception propagates cleanly to the API layer and no cross-context write has been committed

Introducing saga compensation would be over-engineering for this architecture.

## Shell Composition Root — The Physical Coordination Hub

**Location:** `src/shell/composition/`

The shell is the only module permitted to import across component boundaries. It creates and injects the two cross-context adapters:

| Adapter | Implements | Delegates to | Injected into |
|---|---|---|---|
| `DecisionIntelligenceAdapter` | `IDecisionIntelligenceService` (BC-1 port) | `PreSeedContextUseCase` + `ProcessTurnIntelligenceUseCase` (BC-2) | `StartSessionUseCase` and `ProcessTurnUseCase` |
| `KnowledgeRetrievalAdapter` | `IKnowledgeRetrievalService` (BC-2 port) | `ExecuteRetrievalUseCase` (BC-3) | `ProcessTurnIntelligenceUseCase` |

**Wiring:** Container instantiates both adapters and injects them via constructor injection at application startup. BC-1, BC-2, and BC-3 never import each other — each BC sees only its own port interface.

## Summary

The Carton Caps Conversational Assistant has **two cross-context workflows**, both realized without additional coordinator classes:

1. **`SessionInitializationCoordinator`** — embedded in `StartSessionUseCase`. Two contexts (BC-1, BC-2). Sequential. School anchor seeded into BC-2 before session activation.

2. **`ConversationTurnCoordinator`** — embedded in `TurnGraph` (WF-GRAPH-1). Three contexts (BC-1 → BC-2 → BC-3 → BC-2 → BC-1). Stateful pipeline with intent routing, context readiness gates, retrieval retry cycle, and response quality gates.

The **shell composition root** is the physical hub — wiring the two adapters that bridge all context boundaries. The application layer is architecturally complete.
