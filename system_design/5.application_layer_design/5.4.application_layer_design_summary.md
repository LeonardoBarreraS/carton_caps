# Phase 5 — Application Layer Design: Summary

---

## Part I — For Non-Technical Readers

### What problem were we solving?

By the end of Phase 4 we had a precise definition of all the business rules of the Carton Caps conversational assistant: what a session is, what a user preference is, what a retrieval query is, what counts as a valid response. Those rules are locked inside the domain layer and they are deliberately kept pure — they do not know anything about databases, APIs, language models, or the internet.

But rules alone do not make a working system. Something has to answer the question: *when does each rule get invoked, in what order, and who coordinates the whole thing?*

That is exactly what Phase 5 answered.

---

### What is the Application Layer?

Think of the application layer as the **director** of the system.

The domain layer is the group of actors — they each know their own role perfectly. The infrastructure layer is the stage crew — they handle the technical details like databases and AI APIs. The application layer is the director who reads the script, calls each actor at the right moment, and makes sure the whole performance runs in the correct order.

The application layer contains no business rules of its own. Its only job is to coordinate: *load this, call that, confirm this invariant, return the result*.

---

### What does the assistant actually do during a conversation?

Every single message a user sends triggers this exact sequence of steps, orchestrated entirely by the application layer:

1. **The user sends a message.** The system confirms the session is open and marks it as active.
2. **Intent is classified.** The assistant figures out what the user is asking: are they looking for a product? Asking about a referral program? Or is the message ambiguous?
3. **Decision Intelligence builds context.** A dedicated module accumulates what it knows about the user's preferences across turns and checks whether it has enough context to retrieve relevant knowledge.
4. **Clarification is requested if needed.** If the user asked about products but the system has no preference signals yet, it asks a targeted clarifying question instead of guessing.
5. **Knowledge is retrieved.** When context is ready, the system searches the product catalog or referral rules using a semantically enriched query — not just the raw user message.
6. **A response is generated.** The language model receives the user message plus the retrieved evidence and produces a grounded answer.
7. **Quality is evaluated.** The response is scored for groundedness (is it based on real retrieved data?) and relevance (does it actually answer the question?). If the scores are too low, a safe fallback is returned instead.
8. **The turn is recorded.** Both the user message and the assistant response are saved to conversation history.
9. **The session returns to idle.** Ready for the next message.

None of this is visible to the user. From their perspective, they sent a message and received a helpful, grounded answer.

---

### Why does the system have three separate modules?

The three core modules — Conversation Management, Decision Intelligence, and Knowledge Retrieval — each own a distinct responsibility that the others do not touch.

**Conversation Management** is the front door. It knows about sessions, users, schools, and response quality. It does not know how evidence is retrieved or how signals are accumulated — it just asks for a result.

**Decision Intelligence** is the reasoning engine. It knows how to accumulate preferences over turns, detect gaps in context, construct precise retrieval queries, and evaluate the quality of retrieved evidence. It does not know anything about sessions, users, or how responses are generated.

**Knowledge Retrieval** is the library. It holds the product catalog and referral program rules and knows how to search them semantically. It does not know who is asking or why — it only returns the most relevant results for a given query.

This separation means each part can be tested, improved, or replaced independently. If the retrieval strategy changes entirely, only the Knowledge Retrieval module changes. The conversation logic is unaffected.

---

### What do the Shell Adapters do?

The three modules described above are deliberately isolated — they do not import each other's internals. But they still need to communicate at runtime.

The shell adapters are the translators that bridge the modules. When Conversation Management needs Decision Intelligence to process a turn, it does not call Decision Intelligence's code directly — it calls a simple interface (`IDecisionIntelligenceService`). The shell adapter is the concrete object that satisfies that interface: it receives the call, translates the data into the format Decision Intelligence expects, calls the right use case, and translates the result back.

This means the modules stay clean and independent. The only place in the entire system where cross-module coupling is allowed is the shell adapter layer.

---

### What did we actually build?

At the end of Phase 5 the system has all of the following:

| What was built | What it does |
|---|---|
| `StartSessionUseCase` | Opens a session, loads user and school context, pre-seeds the intelligence module |
| `ProcessTurnUseCase` | Orchestrates the full message-to-response cycle using the TurnGraph |
| `CloseSessionUseCase` | Closes a session and enforces that it is idle before closing |
| `PreSeedContextUseCase` | Initialises the Decision Intelligence context with school anchor and purchase history |
| `ProcessTurnIntelligenceUseCase` | Runs the decision subgraph: extract signals → evaluate readiness → retrieve → evaluate evidence |
| `ExecuteRetrievalUseCase` | Routes a query to the product catalog or referral rules and returns the top results |
| TurnGraph (LangGraph) | The stateful pipeline for every conversation turn in BC-1 |
| Decision Intelligence Subgraph (LangGraph) | The stateful reasoning pipeline inside BC-2, including retrieval retry logic |
| `DecisionIntelligenceAdapter` | Shell bridge: BC-1 ↔ BC-2 |
| `KnowledgeRetrievalAdapter` | Shell bridge: BC-2 ↔ BC-3 |

## Part II — Technical Reference

### Layer Role and Constraints

The application layer sits between the domain layer and the infrastructure layer in the Clean Architecture stack.

```
Infrastructure (Phase 6)     — databases, LLM APIs, vector store, REST endpoints
─────────────────────────────────────────────────────────────────────────────────
Application Layer (Phase 5)  — use cases, workflows, coordination, ports
─────────────────────────────────────────────────────────────────────────────────
Domain Layer (Phase 4)       — entities, value objects, invariants, domain events
```

**Rules that govern this layer:**
- Use cases contain no business logic — they orchestrate domain method calls only.
- Every use case is an atomic transaction boundary: load → invoke → persist → drain events → return.
- Domain objects never cross bounded context boundaries. Only primitives and DTOs cross.
- LangGraph graphs are built via factory functions that receive all ports as closure arguments. No domain objects are placed inside graph state.
- The application layer imports the domain layer and defines ports (interfaces) for what it needs from infrastructure. It never imports infrastructure directly.

```
classify_intent
   │
   ├──[product_inquiry | referral_question | general_question]──► process_decision_intelligence

                                │
                        evaluate_response
                                │

               │                │
              END               END
```

- `MIN_GROUNDEDNESS_THRESHOLD = 0.8`
- `MIN_RELEVANCE_THRESHOLD = 0.75`

- `classify_intent` → calls `IIntentClassifier.classify(message)` → sets `intent`

- `generate_clarification` → calls `IResponseGenerator.generate_clarification(message, clarification_question)` → sets `response_text`
- `evaluate_response` → reconstructs `AssistantResponse`, calls `IResponseEvaluator.evaluate()` → sets `groundedness_score`, `relevance_score`
- `persist_turn` → calls `IConversationHistoryRepository.append()` twice (USER + ASSISTANT entries)
- `return_fallback` → calls `AssistantResponse.fallback(reason)` → sets `response_text`, `is_fallback=True`


Graph topology:

```
extract_signals
       │
update_decision_context
       │
evaluate_context_readiness
       │
       ├──[intent in _INTENT_SUFFICIENT_RETRIEVAL]──► construct_retrieval_query
       │    (referral_question: bypass readiness gate)        │
       │                                                execute_retrieval
       ├──[retrieval intent + context ready]──► construct_retrieval_query
       │                                                │
       │                                        execute_retrieval
       │                                                │
       │                                     evaluate_evidence_quality
       │                                                │
       │                           ┌───────────────────┤
       │                           │(recall < threshold │(recall ≥ threshold
       │                           │ AND retries left)  │ OR retries exhausted)
       │                  construct_retrieval_query     END
       │                  (retry back-edge)
       │
       └──[general_question OR clarification_needed]──► END
```

Key design decisions:
- `DecisionContext` is injected as a **closure** — it never enters graph state to prevent serialisation of domain objects.
- **Intent-sufficient retrieval (`_INTENT_SUFFICIENT_RETRIEVAL`):** For `referral_question`, the `evaluate_context_readiness` node bypasses context readiness checks entirely and returns `clarification_needed=False` immediately. `route_after_readiness` mirrors this by routing directly to `construct_retrieval_query` even when `context.is_ready_for_retrieval()` is `False`. This prevents the silent failure that occurred when a user's first referral question produced no evidence because the context was `EMPTY` and no preference signals were extracted from a factual query.
- Gap detection (`_detect_gaps`) is an application-layer function, not a domain method. Rule: for `product_inquiry`, a gap is flagged **only** when the context contains **none** of the five retrieval-useful signal keys (`product_category`, `meal_occasion`, `dietary_restriction`, `health_goal`, `budget_tier`). Any single key from this set is sufficient to produce a meaningful vector-search query, because each key's values appear verbatim in the enriched product documents stored in Qdrant. If at least one retrieval-useful signal is present, no gap is flagged and context proceeds to retrieval. If none are present, a gap covering all five keys is returned and an open-ended clarification question is surfaced to the user. For `referral_question`, this logic is not reached due to the intent-sufficient bypass. The previous design, which flagged a gap whenever `product_category` was absent, created a bottleneck: valid signals such as `meal_occasion=breakfast` or `health_goal=healthy` were ignored as insufficient, forcing unnecessary clarification rounds. The new rule eliminates this bottleneck while preserving the safety invariant that retrieval never runs with a completely empty context.
- Retrieval retry: loop back to `construct_retrieval_query` when `recall < min_recall_threshold AND retrieval_attempts < max_retrieval_retries`. Default thresholds: `min_recall=0.6`, `max_retries=2`. On each retry, `top_k` is escalated by `_TOP_K_INCREMENT` (default +3): first attempt retrieves 5 documents, second retrieves 8, third retrieves 11. This makes each retry a strictly broader search rather than an identical no-op.
- **Vague-message backstop (`_MIN_QUERY_CONTENT_WORDS = 4`):** `construct_retrieval_query` counts words in the assembled `message + signal_values` content. If the count is below the threshold, the current message is an anaphorical reference (`"ok"`, `"tell me more"`, `"and what about?"`) with no retrieval vocabulary. The node then scans `conversation_history` in reverse (windowed to `_LLM_HISTORY_WINDOW`) and injects the most recent substantive prior `user` message as additional query content. Guards: the prior message must differ from the current one and must itself be ≥ `_MIN_QUERY_CONTENT_WORDS` words (to skip chains of vague messages). This is deterministic, zero-cost, and requires no new LLM call.
- `IEvidenceEvaluator` and `ISignalExtractor` use **domain objects** in their signatures — this is intentional and correct since both are intra-BC-2 ports, not cross-context boundaries.

### Shell Adapters

Location: `src/shell/composition/adapters/`

This is the only directory in the entire codebase permitted to import from multiple bounded contexts simultaneously. No domain layer file and no application layer file imports across BC boundaries.

### Cross-Context Call Chain — One User Turn (End to End)

```
API request (user_id, session_id, message)
    │
    ▼
ProcessTurnUseCase.execute(ProcessTurnCommand)
    │  guard: session exists, ownership, status=IDLE
    │  session.receive_message()           [idle → active]
    │
    ▼
TurnGraph.invoke(initial_state)
    │
    ├── classify_intent
    │       └─► IIntentClassifier.classify(message)
    │
    ├── process_decision_intelligence
    │       └─► IDecisionIntelligenceService.process_turn(TurnInputDTO)
    │                   │   [COMM-1 boundary — shell adapter]
    │                   ▼
    │           ProcessTurnIntelligenceUseCase.execute(command)
    │                   │
    │                   ▼
    │           DecisionIntelligenceSubgraph.invoke(state)
    │               ├── extract_signals        → ISignalExtractor
    │               ├── update_decision_context
    │               ├── evaluate_context_readiness
    │               ├── construct_retrieval_query
    │               ├── execute_retrieval
    │               │       └─► IKnowledgeRetrievalService.retrieve(RetrievalQueryDTO)
    │               │                   │   [COMM-2 boundary — shell adapter]
    │               │                   ▼
    │               │           ExecuteRetrievalUseCase.execute(command)
    │               │               └─► IProductRepository.search() or
    │               │                   IReferralRuleRepository.search()
    │               └── evaluate_evidence_quality → IEvidenceEvaluator
    │
    ├── generate_response / generate_clarification
    │       └─► IResponseGenerator.generate() or .generate_clarification()
    │
    ├── evaluate_response
    │       └─► IResponseEvaluator.evaluate()
    │
    ├── persist_turn
    │       └─► IConversationHistoryRepository.append() × 2
    │
    └── return_fallback (only if quality threshold not met)
    │
    ▼
session.complete_turn(intent, response_vo)  [active → idle]
IConversationSessionRepository.save(session)
session.collect_events()
    │
    ▼
ProcessTurnResponse(session_id, response_text, intent, is_fallback, scores)
```

### What Phase 5 Does Not Include

The following are intentionally deferred to Phase 6 (Infrastructure):
- Concrete `IIntentClassifier`, `IResponseGenerator`, `IResponseEvaluator` implementations (LLM adapters)
- Concrete `ISignalExtractor`, `IEvidenceEvaluator` implementations
- `IProductRepository`, `IReferralRuleRepository` implementations (vector store adapters)
- `IConversationSessionRepository`, `IConversationHistoryRepository` implementations (database adapters)
- REST API controllers and entry points
- Composition root (the code that instantiates all adapters, repositories, and use cases at startup)

All of these depend on the application layer contracts defined in this phase. Phase 5 freezes those contracts so Phase 6 can implement them without touching domain or application logic.
