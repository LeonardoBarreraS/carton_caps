# Workflow Orchestration — Carton Caps Conversational Assistant

**Phase:** 5 — Application Layer Design  
**Task:** Workflow Orchestration  
**Derived from:** `use_cases.json`, `domain_model.json`, `context_communication.json`, `components.json`

---

## Design Principles

- Workflows are **execution engines** — they orchestrate domain behavior, not implement domain rules.
- **Domain objects are never in graph state.** `ConversationSession` and `DecisionContext` are loaded by the parent use case, passed to node functions as injected dependencies, and saved by the use case after the graph returns.
- **Graph state carries only primitives and read-only snapshots** — typed scalars, lists of strings, serialized signal dicts.
- **Nodes call ports only** — no LLM calls, no DB access inline. All I/O is behind interfaces.
- Conditional edges encode **application-level routing decisions** driven by domain state values (intent, context readiness, quality scores).
- **The parent use case owns the transaction boundary.** It loads before, saves after. The graph handles the pipeline in between.

---

## Which Use Cases Need Workflow Orchestration

| Use Case | Workflow? | Rationale |
|---|---|---|
| `StartSessionUseCase` (UC-1) | No | Sequential imperative flow — load, create, seed, save |
| `ProcessTurnUseCase` (UC-2) | **Yes — TurnGraph** | Multi-step conditional pipeline: classify → decide → generate → evaluate → persist/fallback |
| `CloseSessionUseCase` (UC-3) | No | Single domain method call + save |
| `PreSeedContextUseCase` (UC-4) | No | Single domain creation + optional signal seeding + save |
| `ProcessTurnIntelligenceUseCase` (UC-5) | **Yes — DecisionIntelligenceSubgraph** | Multi-step pipeline with gap evaluation, context readiness branching, and retrieval retry cycle |
| `ExecuteRetrievalUseCase` (UC-6) | No | Simple routing switch: route to one of two repositories |

Only UC-2 and UC-5 have pipelines complex enough to justify LangGraph execution engines. Everything else remains as clean imperative use case code.

## WF-GRAPH-1: TurnGraph

**Module:** `src/conversation_management/application/workflows/turn_graph.py`  
**Parent use case:** `ProcessTurnUseCase` (UC-2)  
**Component:** BC-1 — conversation_management  
**Workflow:** WF0 — per-turn execution

**Purpose:** Executes the full conversation turn pipeline. Classifies intent, routes no-retrieval intents (`out_of_scope`, `general_question`) directly to a unified `generate_redirect` node that bypasses Decision Intelligence entirely, delegates RAG intents to Decision Intelligence for evidence retrieval, generates a grounded response, evaluates quality scores, and routes to either turn persistence (success) or `generate_redirect` (quality failure).

---

All fields are primitives — no domain objects in state.

| `user_name` | `str` | use case (init) — loaded from `IUserRepository.find_by_id(user_id).name`; forwarded to N3 `context_snapshot`, N4 and N7 `user_context` dict for LLM personalization |

| `relevance_score` | `float` | `evaluate_response` |

### Nodes

#### `classify_intent`
- **Reads:** `message`
- **Writes:** `intent`
- **Calls:** `IIntentClassifier.classify(message)` → `Intent`
- **Enforces:** INV-WF-1 — intent classified before context update and retrieval

#### `process_decision_intelligence`
- **Reads:** `session_id`, `intent`, `message`, `turn_number`, `conversation_history`
- **Writes:** `evidence_chunks`, `evidence_type`, `context_recall_score`, `context_precision_score`, `context_state`, `clarification_needed`, `clarification_question`
- **Calls:** `IDecisionIntelligenceService.process_turn(TurnInputDTO)` → `TurnIntelligenceResultDTO`
  - `TurnInputDTO` includes windowed `conversation_history` (last `_LLM_HISTORY_WINDOW` = 5 messages) so BC-2 provides recent conversational context to LLM calls without unbounded token growth
  - Cross-context call → `DecisionIntelligenceAdapter` (shell) → `ProcessTurnIntelligenceUseCase` (BC-2)
- **Enforces:** INV-WF-3 (retrieval complete before generation), INV-WF-4 (DI runs on every turn)

#### `generate_response`
- **Reads:** `message`, `evidence_chunks`, `evidence_type`, `context_state`, `user_name`, `school_name`
- **Writes:** `response_text`
- **Calls:** `IResponseGenerator.generate(message, evidence, context_snapshot)` → `AssistantResponse`
  - `context_snapshot` includes `user_name` (from state, default `"there"`) and `school_name`
  - **Only reached for RAG intents** (`product_inquiry`, `referral_question`, `clarification_response`) — `general_question` is now routed directly to `generate_redirect` at N1, bypassing DI and generation entirely
  - LLM system prompt personalised with `user_name` and `school_name` via `_SYSTEM_PROMPT_TEMPLATE`
- **Enforces:** INV-AR-2 (product evidence only), INV-AR-3 (referral rules only), INV-AR-4 (school reference in prompt)

#### `generate_clarification`
- **Reads:** `message`, `intent`, `clarification_question`, `user_name`, `school_name`
- **Writes:** `response_text`
- **Calls:** `IResponseGenerator.generate_clarification(message, clarification_question, user_context)` → `AssistantResponse`
  - `user_context` = `{"user_name": state["user_name"] or "there", "school_name": state["school_name"] or "your school"}` — built inline by this node
  - If `clarification_question` is set (from BC-2 gap detection): wraps it into a personalised response
  - If intent is `ambiguous` (no question from BC-2): LLM generates an appropriate clarification
- **Note:** `ambiguous` intent routes here. `clarification_response` intent routes to `process_decision_intelligence` so the DI service updates context with the user's answer.

#### `evaluate_response`
- **Reads:** `response_text`, `intent`, `evidence_chunks`
- **Writes:** `groundedness_score`, `relevance_score`
- **Calls:** `IResponseEvaluator.evaluate(response, intent, evidence)` → scored response
  - `groundedness_score`: claim traceability to retrieved evidence
  - `relevance_score`: reverse-question technique (does the response actually answer the question?)
- **Domain method:** `AssistantResponse.meets_quality_threshold(min_groundedness, min_relevance)` — routing decision
- **Note:** Only reached for RAG intents (`product_inquiry`, `referral_question`, `clarification_response`). `general_question` exits at N1 via `generate_redirect`, never reaching this node. `generate_clarification` routes directly to `persist_turn`.
- **Enforces:** INV-AR-1, INV-AR-5, INV-AR-6, INV-WF-6

#### `persist_turn`
- **Reads:** `session_id`, `user_id`, `message`, `intent`, `response_text`, `turn_number`
- **Calls:** `IConversationHistoryRepository.append()` × 2
  - `ConversationHistory.create(..., sender=user)` — user message entry
  - `ConversationHistory.create(..., sender=assistant)` — assistant response entry
- **Enforces:** INV-WF-1 — turn persisted to history on completion

#### `generate_redirect`
- **Reads:** `message`, `intent`, `user_name`, `school_name`, `conversation_history`, `evidence_chunks`, `evidence_type`
- **Writes:** `response_text`
- **Calls:** `IResponseGenerator.generate_redirect(message, conversation_history, user_context, redirect_reason, evidence)` → `AssistantResponse`
  - `redirect_reason` derived from `state["intent"]`:
    - `"out_of_scope"` — intent was classified as `out_of_scope` at N1
    - `"general_question"` — intent was `general_question`; DI entirely bypassed
    - `"low_quality"` — RAG intent that failed quality evaluation in `evaluate_response`
  - For `"low_quality"`: `evidence = {"chunks": state["evidence_chunks"], "evidence_type": state["evidence_type"]}` is passed to the port so the adapter injects the already-retrieved chunks into a dedicated `_LOW_QUALITY_WITH_EVIDENCE_SYSTEM` prompt that forbids deflection, instructs the LLM to answer from the partial evidence, and requires one focused follow-up question
  - For `"out_of_scope"` and `"general_question"`: `evidence=None`; the adapter uses `_REDIRECT_SYSTEM` with `_REDIRECT_REASON_INSTRUCTIONS`
  - `user_context` = `{"user_name": state["user_name"] or "there", "school_name": state["school_name"] or "your school"}`
  - Scores hardcoded to `1.0` — quality gate is not applied to redirect responses
- **Routes:** unconditionally to `persist_turn`
- **Enforces:** INV-AR-5 — quality-gate failures handled here; low-quality responses are never delivered

### Conditional Edges

```
classify_intent
  ├─ intent == out_of_scope                                        →  generate_redirect  (redirect_reason='out_of_scope'; DI bypassed)
  ├─ intent == general_question                                     →  generate_redirect  (redirect_reason='general_question'; DI bypassed)
  ├─ intent == ambiguous                                            →  generate_clarification
  └─ intent in [product_inquiry, referral_question,                 →  process_decision_intelligence
               clarification_response]

process_decision_intelligence
  ├─ clarification_needed == True   →  generate_clarification
  └─ clarification_needed == False  →  generate_response

generate_response                      →  evaluate_response  (always — only RAG intents reach this node)

generate_clarification                 →  persist_turn  (clarification questions are not graded against evidence)

evaluate_response
  ├─ groundedness >= threshold AND relevance >= threshold  →  persist_turn
  └─ groundedness <  threshold OR  relevance <  threshold  →  generate_redirect  (redirect_reason='low_quality')

generate_redirect                      →  persist_turn  (always)

persist_turn                           →  END
```

### Post-Graph Use Case Actions (after TurnGraph returns)

These happen in `ProcessTurnUseCase`, not inside the graph:
1. `session.complete_turn(intent, response)` — transitions `active → idle`; emits `TurnProcessed`; increments `turn_count`
2. `IConversationSessionRepository.save(session)` — persists session state
3. Drain domain events (`TurnProcessed`)

## WF-GRAPH-2: DecisionIntelligenceSubgraph

**Module:** `src/decision_intelligence/application/workflows/decision_intelligence_subgraph.py`  
**Parent use case:** `ProcessTurnIntelligenceUseCase` (UC-5)  
**Component:** BC-2 — decision_intelligence  
**Workflows:** WF1 (Decision Context Building) + WF2 (Product Recommendation) + WF3 (Referral Guidance)

**Purpose:** Executes the full decision intelligence pipeline per turn. Extracts preference signals, updates `DecisionContext`, evaluates context readiness, constructs an enriched retrieval query, executes retrieval via COMP-3, and evaluates evidence quality. Includes the retrieval retry cycle when recall falls below threshold.

---

### Domain Object Handling

`DecisionContext` is **not** in graph state. It is:
1. Loaded by `ProcessTurnIntelligenceUseCase` before graph invocation
2. Passed to node functions as a **captured dependency** (closure)
3. Mutated through domain methods (`add_signal()`, `record_gap_evaluation()`, `mark_ready()`) inside relevant nodes
4. Saved by the use case **after** the graph returns

This preserves Clean Architecture: domain objects stay outside framework primitives.

---

| Field | Type | Set by |

| `intent` | `str` | use case (init); may be overridden to `pending_retrieval_intent` by `evaluate_context_readiness` when resuming a clarification loop |
| `message` | `str` | use case (init) |
| `turn_number` | `int` | use case (init) |
| `conversation_history` | `list[dict]` | use case (init) — forwarded from `TurnInputDTO`; used by `extract_signals` — **sliced to last `_LLM_HISTORY_WINDOW` (5) messages by `_recent()` helper inside the node before any LLM call** |
| `context_snapshot` | `dict` | use case (init via `context.snapshot()`); updated after `update_decision_context` |
| `extracted_signals` | `list[dict]` | `extract_signals` — serialized `{key, value, turn_number}` |

| `context_precision_score` | `float` | `evaluate_evidence_quality` |
| `retrieval_attempts` | `int` | incremented by `execute_retrieval` on each attempt |
| `context_state` | `str` | `update_decision_context` — current `DecisionContext.status` as string |

### Nodes

#### `extract_signals`
- **Reads:** `message`, `turn_number`, `conversation_history`
- **Writes:** `extracted_signals`
- **Calls:** `ISignalExtractor.extract(message, turn_number, conversation_history)` → `list[PreferenceSignal]`
  - LLM adapter extracts structured preference signals (key-value pairs) from free-text user message
  - `conversation_history` is windowed to the last `_LLM_HISTORY_WINDOW` (5) messages by the `_recent()` helper before being passed, providing sufficient recent context for accurate signal extraction while controlling token cost
  - Output serialized as list of dicts for graph state
- **Enforces:** INV-WF-1 (signals extracted before context update), INV-WF-4 (always runs)

#### `update_decision_context`
- **Reads:** `extracted_signals`
- **Writes:** `context_snapshot`, `context_state`
- **Domain methods called:**
  - `PreferenceSignal(key, value, turn_number)` — re-hydrate from serialized dict
  - `context.add_signal(signal)` — for each signal; domain enforces monotonic accumulation; may emit `ContextUpdated`
  - `context.snapshot()` → dict — capture updated state for routing
- **Enforces:** INV-DC-5 (monotonic accumulation), INV-WF-2 (context updated before query construction)

#### `evaluate_context_readiness`
- **Reads:** `context_snapshot`, `intent`
- **Writes:** `gaps`, `clarification_needed`, `clarification_question`, `context_state`; may write `intent` (see below)
- **Domain methods called:**
  - `context.has_gap()` → bool
  - `context.record_gap_evaluation(gaps)` — may emit `GapDetected`
  - `context.mark_ready()` — if status == partial and signals sufficient; emits `ContextReady`
  - `context.is_ready_for_retrieval()` → bool — routing decision
  - `context.set_pending_retrieval_intent(intent)` — saves original retrieval intent when a gap is detected (so it can be resumed after the user answers the clarification)
  - `context.clear_pending_retrieval_intent()` — clears saved intent once retrieval is resumed successfully
- **`clarification_response` resume logic:** When `intent == "clarification_response"`, the node reads `context.pending_retrieval_intent` to recover the original request. If a pending intent exists, the node runs the full readiness check against it and may override `state["intent"]` to the pending intent so that `construct_retrieval_query` and `route_after_readiness` use the correct source target. If no pending intent exists, the node exits silently with `clarification_needed=False`.
- **Gap evaluation note:** Gaps are derived from the extracted signal set and intent using `_detect_gaps()` backed by `_RETRIEVAL_USEFUL_SIGNAL_KEYS = {product_category, meal_occasion, dietary_restriction, health_goal, budget_tier}`. For `product_inquiry`, a gap is emitted only when the context contains **zero** signals from this set — meaning not a single retrieval-useful key has been captured. Any one of these five keys is sufficient for a meaningful vector-search query. When a gap is detected, `_build_clarification_question()` produces an open-ended conversational prompt (not a list of raw key names) that guides the user toward providing at least one useful signal. `pending_retrieval_intent` is saved on the context so the loop can resume after the user answers. For `referral_question`, gap detection is bypassed entirely — see the intent-sufficient note below.
- **Intent-sufficient retrieval bypass (`_INTENT_SUFFICIENT_RETRIEVAL`):** For intents in `_INTENT_SUFFICIENT_RETRIEVAL = {"referral_question"}`, the node returns immediately with `clarification_needed=False`, skipping `is_ready_for_retrieval()`, gap detection, and `mark_ready()` entirely. The user's question text is the retrieval signal; no preference signals are ever needed to query the referral rule store. `route_after_readiness` mirrors this: it routes directly to `construct_retrieval_query` for any intent in `_INTENT_SUFFICIENT_RETRIEVAL`, even when `context.is_ready_for_retrieval()` is `False` (e.g. context is `EMPTY` on the first turn of a session). This fixes the silent failure where a first-turn referral question produced no evidence because the context readiness gate blocked retrieval before any signals could be accumulated.
- **Enforces:** INV-DC-4 (retrieval only if ready/enriched — bypassed for intent-sufficient intents), INV-DC-6 (mark_ready guards signal count), INV-PRI-1 (pending retrieval intent preserved across clarification turns)

#### `construct_retrieval_query`
- **Reads:** `session_id`, `intent`, `message`, `context_snapshot`, `retrieval_attempts`, `conversation_history`
- **Writes:** `retrieval_query_text`, `source_target`, `top_k`
- **Domain methods called:**
  - `RetrievalQuery.from_context(session_id, query_text, intent, context_status)` — factory enforces INV-RQ-2, INV-RQ-3
- **Query construction strategy:**
  - For fresh intents (`product_inquiry`, `referral_question`): the user's **message is the primary query component** — it carries the exact vocabulary that appears in the vector store (e.g. `"eligibility"`, `"how to refer"`, `"healthy breakfast cereal"`). Signal values are appended as reinforcement. School name is excluded — it does not exist in product or referral chunk embeddings.
  - For `clarification_response` resumes (where the message is a vague answer like `"yes"` or `"cereal"`): signal values are the primary query content. The message is skipped.
  - A minimal domain anchor (`"fundraising product"` or `"referral program"`) is appended to anchor the embedding in the right semantic neighbourhood.
- **Vague-message backstop (`_MIN_QUERY_CONTENT_WORDS = 4`):** After assembling `message + signal values`, if the combined content is fewer than `_MIN_QUERY_CONTENT_WORDS` words, the current message is an anaphorical reference (`"ok"`, `"tell me more"`, `"and that?"`) that carries no retrieval vocabulary. The node then walks `conversation_history` in reverse (windowed to the last `_LLM_HISTORY_WINDOW` entries) and injects the most recent substantive prior `user` message (one that is different from the current message and is itself ≥ `_MIN_QUERY_CONTENT_WORDS` words) into `parts`. This resolves the implicit reference deterministically with no LLM call. Both `product_inquiry` (where typically signals cover the gap if any are present) and `referral_question` (where signals are never generated) benefit from this mechanism.
- **top_k escalation on retry:** `top_k = _BASE_TOP_K + retrieval_attempts × _TOP_K_INCREMENT` (defaults: base=5, increment=3). Each retry retrieves a strictly larger result set — first attempt: 5, second: 8, third: 11. This makes every retry genuinely different from the previous one.
- **Enforces:** INV-RQ-1 (query derived from context + message, not passed as raw unprocessed input), INV-RQ-2, INV-RQ-3

#### `execute_retrieval`
- **Reads:** `retrieval_query_text`, `source_target`, `top_k`, `retrieval_attempts`
- **Writes:** `evidence_chunks`, `evidence_type`, `retrieval_attempts` (incremented)
- **Calls:** `IKnowledgeRetrievalService.retrieve(RetrievalQueryDTO)` → `RetrievedEvidenceDTO`
  - `RetrievalQueryDTO` now carries `top_k` which is forwarded through `KnowledgeRetrievalAdapter` → `ExecuteRetrievalCommand` → `ExecuteRetrievalUseCase`
  - Cross-context call → `KnowledgeRetrievalAdapter` (shell) → `ExecuteRetrievalUseCase` (BC-3)
- **Enforces:** INV-WF-3 (retrieval before generation)

#### `evaluate_evidence_quality`
- **Reads:** `retrieval_query_text`, `source_target`, `evidence_chunks`, `evidence_type`
- **Writes:** `context_recall_score`, `context_precision_score`
- **Calls:** `IEvidenceEvaluator.evaluate(query, evidence)` → scored evidence
  - Returns evidence enriched with `context_recall_score` and `context_precision_score`
- **Domain methods:** `RetrievedEvidence.meets_recall_threshold(min_recall)` — routing decision
- **Enforces:** INV-RE-1, INV-RE-2, INV-WF-5 (quality evaluated before evidence reaches generation)

### Conditional Edges

```
extract_signals            →  update_decision_context   (always)
update_decision_context    →  evaluate_context_readiness (always)

evaluate_context_readiness
  ├─ intent == clarification_response AND pending_retrieval_intent exists AND context ready  →  construct_retrieval_query (resumes original intent)
  ├─ intent == clarification_response AND pending_retrieval_intent exists AND gaps remain    →  END  (clarification_needed=True; pending intent kept)
  ├─ intent == clarification_response AND no pending_retrieval_intent                        →  END  (nothing to resume)
  ├─ intent in _INTENT_SUFFICIENT_RETRIEVAL (referral_question)                             →  construct_retrieval_query (message IS the query; context readiness gate bypassed)
  ├─ intent requires_retrieval AND context.is_ready_for_retrieval()                          →  construct_retrieval_query (INV-DC-4 satisfied)
  ├─ intent requires_retrieval AND NOT ready AND clarification_needed                        →  END  (clarification_needed=True; pending_retrieval_intent saved)
  └─ intent in [general_question]                                                            →  END  (no retrieval, empty evidence)

construct_retrieval_query  →  execute_retrieval          (always)
execute_retrieval          →  evaluate_evidence_quality  (always)

evaluate_evidence_quality
  ├─ recall >= MIN_RECALL_THRESHOLD                                              →  END  (quality passes, evidence returned)
  ├─ recall <  MIN_RECALL_THRESHOLD AND retrieval_attempts < MAX_RETRIES         →  construct_retrieval_query  (INV-RE-3: retry with refined query)
  └─ recall <  MIN_RECALL_THRESHOLD AND retrieval_attempts >= MAX_RETRIES        →  END  (retries exhausted, degraded evidence flagged)
```

**Clarification resume loop:** When `intent == "clarification_response"` arrives, `evaluate_context_readiness` reads `context.pending_retrieval_intent` (saved in a prior turn). If found and context is now ready, `state["intent"]` is overridden to the original intent (e.g. `"product_inquiry"`) so `construct_retrieval_query` and `route_after_readiness` behave identically to a fresh retrieval turn.

**Retry cycle note:** On retry, `construct_retrieval_query` produces a genuinely different result via two mechanisms: (1) the same message+signal query text is submitted again, and (2) `top_k` is escalated by `_TOP_K_INCREMENT` (default: +3) so the retrieval window is wider. The `retrieval_attempts` counter bounds the retry cycle to `MAX_RETRIEVAL_RETRIES` (default: 2, configurable).

---

### Exit Points Summary

| Exit path | Meaning |
|---|---|
| `evaluate_context_readiness → END` (general/clarification) | No retrieval needed. Empty evidence returned to BC-1. |
| `evaluate_context_readiness → END` (gap detected) | Context insufficient. `clarification_needed=True` returned to BC-1. |
| `evaluate_evidence_quality → END` (quality passes) | Evidence ready. Good recall/precision scores returned with chunks. |
| `evaluate_evidence_quality → END` (retries exhausted) | Degraded evidence. Low recall score returned. BC-1 evaluate_response will likely fallback. |

## Full Turn Execution Picture

```
ProcessTurnUseCase
  ├─ load session; session.receive_message()   [UC-2 pre-graph]
  │
  ├─ TurnGraph.invoke(TurnGraphState)          [WF-GRAPH-1]
  │    classify_intent
  │    └─ process_decision_intelligence
  │         └─ (cross-context) → ProcessTurnIntelligenceUseCase
  │              └─ DecisionIntelligenceSubgraph.invoke(DecisionGraphState)   [WF-GRAPH-2]
  │                   extract_signals
  │                   update_decision_context
  │                   evaluate_context_readiness
```
