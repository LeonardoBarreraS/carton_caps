# Phase 5: Application Layer Design — Analysis

## Why this phase exists at all

The domain layer (Phase 4) defines *what is valid* — rules, invariants, state transitions. But it has zero idea of *when things happen*, *in what order*, or *how BCs interact*. Phase 5 fills that gap: it wires the domain into executable system behavior.

Without Phase 5, you'd either dump orchestration logic into infrastructure adapters (controllers, LangGraph nodes), or mix it into the domain itself — both violate Clean Architecture.

---

## The three tasks form a deliberate hierarchy

The skills are ordered by **scope of coordination**:

| Task | Scope | What it creates |
|---|---|---|
| Use Case Generator | 1 BC / 1 action | `UseCase.execute(command)` |
| Workflow Orchestration | 1 workflow / multi-aggregate | `WorkflowOrchestrator` inside a BC |
| Context Coordination | System-wide / multi-BC | `CrossContextCoordinator` |

They compose: a coordinator calls use cases, a use case may delegate to a workflow orchestrator, the orchestrator calls domain methods.

---

## Task 1 — Use Case Generator: Essential

Creates the **entry points** into each BC. For this system:

- `StartSessionUseCase` — BC-1
- `ProcessTurnUseCase` — BC-1
- `BuildDecisionContextUseCase` — BC-2
- `QueryKnowledgeBaseUseCase` — BC-3

Each use case does exactly: load → invoke domain → persist → drain events → return. No business logic. This is also the **transaction boundary** — each use case is one atomic operation.

---

## Task 2 — Workflow Orchestration: Valuable but nuanced

This is for multi-step flows *within a BC or across aggregates*. For this project, the most important example is the **BC-1 session lifecycle**:

```
begin_initialization()
  → (call LLM for profile)
complete_initialization()
  → session now ACTIVE
receive_message()
complete_turn(intent, response)
```

These steps can't be collapsed into one use case — they have intermediate state (the `INITIALIZING` status on `ConversationSession`). A `SessionInitializationWorkflow` orchestrator manages this sequence without polluting use cases with multi-step logic.

---

## Task 3 — Context Coordination: The most critical task for this system

This project's core flow — processing a conversational turn — crosses **all three BCs**:

```
1. BC-1: receive_message(), classify intent
2. BC-2: add_signal(), record_gap_evaluation(), mark_ready()
3. BC-3: search products / search referral rules
4. BC-2: evaluate evidence (IEvidenceEvaluator)
5. BC-1: generate response (IResponseGenerator), evaluate quality, complete_turn()
```

A single `ConversationTurnCoordinator` (in `application/coordination/`) would orchestrate this entire sequence — calling use cases from each BC in order, passing primitive data across (never domain objects), advancing the global conversation workflow.

This is also likely where **LangGraph** materializes: the coordinator's state machine becomes a LangGraph graph, with conditional edges based on `intent.requires_retrieval()`, `context.is_ready_for_retrieval()`, and `response.meets_quality_threshold()`.

---

## Does Phase 5 fully materialize the application layer?

**Yes — completely.** After these three tasks, you have:

```
src/
  conversation_management/application/
    use_cases/          ← Task 1
    workflows/          ← Task 2
    coordination/       ← Task 3 (or shared shell layer)
  decision_intelligence/application/
    use_cases/
    workflows/
  knowledge_retrieval/application/
    use_cases/
```

The infrastructure layer (repositories, LLM adapters, vector store adapters, API entry points) comes in Phase 6 — but those phases can only be implemented correctly once the application layer contracts (ports + use cases + coordinators) are frozen from Phase 5.

---

## Overall Opinion

All three tasks are genuinely necessary for **this specific system** — it's not a CRUD app where one use case per entity suffices. The conversation turn is a multi-step, multi-BC, stateful workflow where getting the orchestration right is architecturally the hardest part. The three-level decomposition (use case → workflow → coordinator) prevents that complexity from collapsing into one giant unmaintainable function.
