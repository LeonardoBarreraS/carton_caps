# Bounded Context Detection
## Carton Caps Conversational Assistant

---

## Overview

Three bounded contexts emerge from this domain. Each has a distinct semantic purpose, clear ownership, and non-overlapping invariants.

| Context | Core Responsibility |
|---|---|
| Conversation Management | Session lifecycle and turn orchestration |
| Decision Intelligence | Preference accumulation and retrieval query construction |
| Knowledge Retrieval | Knowledge sources and retrieval execution |

`School` is a shared reference — owned by Conversation Management (loaded from DB at session init) and passed into Decision Intelligence as a pre-seeded signal. It is not an entity inside Knowledge Retrieval.

---

## Context: Conversation Management

### Owns
- The full lifecycle of a user session from creation to close
- Every conversation turn — sequencing, persistence, response delivery

### Entities
- `ConversationSession` (process entity — primary lifecycle owner)
- `ConversationHistory` (immutable historical entity — append-only log of persisted message records, loaded at session init)

### Value Objects
- `User` (external reference — identity reference to User entity in external Auth BC, pre-resolved, never authenticated or mutated here)
- `School` (external reference — immutable identity reference, loaded from DB at session init, passed to Decision Intelligence)
- `Intent` (classified per turn; drives routing to Decision Intelligence and Knowledge Retrieval)
- `AssistantResponse` (grounded response returned to caller)

### Invariants
- `INV-CS-1` through `INV-CS-7` — session state validity, transition rules, single-session constraint
- `INV-WF-1` — intent classified before context update
- `INV-WF-3` — retrieval complete before response generation
- `INV-WF-4` — WF1 (context building) executes on every turn without exception

### Responsibilities
- Manage session lifecycle (`created → initializing → idle ⇄ active → closed`)
- Load user profile, school, conversation history, and purchase history at session start
- Classify intent on every user message before any other processing
- Sequence conversation turns and ensure ordering invariants hold
- Persist completed turns to `ConversationHistory`
- Assemble and return the grounded `AssistantResponse` to the caller

## Context: Decision Intelligence

### Owns
- The progressive accumulation of user preference signals across turns
- The construction of every retrieval query from context — never from the raw message

### Entities
- `DecisionContext` (process entity — secondary lifecycle, owned within a session)

### Value Objects
- `RetrievalQuery` (immutable historical artifact; constructed from context + intent; interface to Knowledge Retrieval)
- `PreferenceSignal` (a single extracted preference; immutable once created; accumulates monotonically)
- `RetrievedEvidence` (received from Knowledge Retrieval; injected into response generation)

### Invariants
- `INV-DC-1` through `INV-DC-6` — context state validity, no-regression rule, monotonic accumulation, school always present
- `INV-RQ-1` through `INV-RQ-3` — retrieval query always built from context + intent, valid source routing
- `INV-WF-2` — context updated before retrieval query constructed

### Responsibilities
- Extract `PreferenceSignal` values from each user message
- Accumulate signals into `DecisionContext` across turns (monotonically — never regresses)
- Evaluate context gaps and determine when context reaches `ready` state
- Emit clarification signal when gaps prevent recommendation
- Construct `RetrievalQuery` from `intent + accumulated DecisionContext` (never from raw message)
- Route the query to the correct knowledge source: `product_catalog` or `referral_program_rules`

---

## Context: Knowledge Retrieval

### Owns
- All knowledge sources available to the assistant
- The execution of semantic retrieval against those sources

### Entities

### Value Objects
- `Product` (external reference value object; retrieved by relevance, never invented)
- `ReferralRule` (external reference value object; platform-defined rule, retrieved by relevance, never rewritten)
- `RetrievedEvidence` (the result of retrieval; passed back to Decision Intelligence for injection)

### Invariants
- `INV-AR-1` — every response grounded in retrieved evidence; ungrounded LLM responses not permitted
- `INV-AR-2` — product recommendations cite only retrieved products
- `INV-AR-3` — referral guidance derived only from retrieved referral rules
- `INV-AR-4` — every product recommendation references the user's school fundraising goal

### Responsibilities
- Maintain the product catalog as a semantically searchable knowledge source
- Maintain referral program rules as a semantically searchable knowledge source
- Execute semantic retrieval for a given `RetrievalQuery`
- Return top-matching `RetrievedEvidence` to Decision Intelligence

## Context Map

```
Conversation Management
        │
        │ per-turn: classified Intent + session state
        ▼
Decision Intelligence
        │
        │ RetrievalQuery (intent + accumulated context)
        ▼
Knowledge Retrieval
        │
        │ RetrievedEvidence (top-matching results)
        ▼
Decision Intelligence → passes evidence to →
Conversation Management (assembles grounded AssistantResponse)
```

### Interaction Description

| From | To | Payload | When |
|---|---|---|---|
| Conversation Management | Decision Intelligence | classified `Intent`, current turn message | Every turn |
| Decision Intelligence | Knowledge Retrieval | `RetrievalQuery` (context-enriched) | When context is `partial`/`ready`/`enriched` with `product_inquiry` or `referral_question` intent |
| Knowledge Retrieval | Decision Intelligence | `RetrievedEvidence` | After retrieval executes |
| Decision Intelligence | Conversation Management | `RetrievedEvidence` | For injection into response generation |

### RAG Pipeline as Context Flow

This project's core RAG pipeline maps directly onto the context boundaries:

1. **Conversation Management** classifies intent and activates the turn
2. **Decision Intelligence** builds the retrieval query from accumulated context
3. **Knowledge Retrieval** executes the query and returns evidence
4. **Decision Intelligence** passes evidence back
5. **Conversation Management** assembles the grounded response and returns it to the caller

The retrieval query never leaves Decision Intelligence as a raw user message — this is enforced by `INV-RQ-1`.
