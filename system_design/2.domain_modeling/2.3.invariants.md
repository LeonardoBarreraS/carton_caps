# Invariant Detection
## Carton Caps Conversational Assistant

---

## Process Entity: ConversationSession

### State Invariants

- Status must be one of: `created`, `initializing`, `active`, `idle`, `closed`
- A session cannot enter `active` state before initialization is complete
- `closed` is a terminal state — no further transitions are possible
- Every session in `active` state must have exactly one associated `DecisionContext`
- Every request to the system must carry a pre-resolved, trusted `user_id`

### Transition Invariants

Allowed:
- `created` → `initializing`
- `initializing` → `idle`
- `idle` → `active`
- `active` → `idle` via `complete_turn()` on success
- `active` → `idle` via `fail_turn()` on error (rollback — no event emitted; user can retry)
- `idle` → `closed`

Not Allowed:
- `created` → `active` (initialization cannot be skipped)
- `closed` → any (terminal — no recovery)
- `active` → `closed` (must complete the current turn and return to idle first)
- any → `initializing` (re-initialization is not permitted during a session)

### Dependency Invariants

- The user's school must be resolved and pre-seeded into `DecisionContext` during `initializing`, before the first turn begins
- A turn cannot be processed while the session is in `initializing` state

### Consistency Invariants

- Every session must have a `school` value present in its `DecisionContext` before any response is generated
- A user cannot have more than one session in `active` or `idle` state simultaneously

---

## Process Entity: DecisionContext

### State Invariants

- Status must be one of: `empty`, `partial`, `ready`, `enriched`
- The `school` field must always be present — it is never null or missing
- The `school` field is set at session initialization and never changes during the session

### Transition Invariants

Allowed:
- `empty` → `partial`
- `partial` → `partial` (more signals added, gaps remain)
- `partial` → `ready`
- `ready` → `enriched`
- `enriched` → `enriched`

Not Allowed:
- `ready` → `partial` (context never regresses)
- `ready` → `empty` (context never regresses)
- `enriched` → `ready` (context never regresses)
- `enriched` → `partial` (context never regresses)
- `enriched` → `empty` (context never regresses)
- `empty` → `ready` (must accumulate at least one signal first)

### Dependency Invariants

- Product retrieval (WF2) cannot activate unless intent is `product_inquiry` OR `DecisionContext` is `ready` or `enriched`
- A `RetrievalQuery` cannot be constructed from an `empty` context without a classified intent

### Consistency Invariants

- Preference signals accumulate monotonically — once added, a signal is never removed
- `identified_gaps` must be cleared when context transitions to `ready`
- A context in `ready` or `enriched` state must have at least one `PreferenceSignal` beyond the pre-seeded school
- `pending_retrieval_intent` must only be set to a value from `_RETRIEVAL_INTENTS` (`product_inquiry`, `referral_question`) — never to `general_question`, `clarification_response`, `ambiguous`, or `out_of_scope` (INV-PRI-1)
- `pending_retrieval_intent` must be cleared atomically when retrieval is resumed — it must not persist across a successful retrieval turn (INV-PRI-2)

---

## Value Object: RetrievalQuery

### Consistency Invariants

- A retrieval query must always be constructed from `intent + accumulated DecisionContext` — never from the raw user message alone
- `source_target` must be one of: `product_catalog`, `referral_program_rules`
- `source_target` must match intent routing: `product_inquiry` → `product_catalog`; `referral_question` → `referral_program_rules`

---

## Value Object: AssistantResponse

### Consistency Invariants

- Every response must be grounded in retrieved evidence — pure LLM-generated responses without retrieval are not permitted. `groundedness_score` must meet the minimum threshold (e.g., ≥ 0.8) before the response is returned to the caller.
- Product recommendations must cite only products returned by the retrieval step — the LLM cannot invent products
- Referral guidance must be derived exclusively from retrieved referral rules — not from LLM memory
- Every product recommendation must explicitly reference the user's school fundraising goal
- If `groundedness_score` or `relevance_score` falls below the minimum threshold, the response must not be returned. A safe fallback message is delivered instead.
- `relevance_score` measures intent alignment using the reverse-question technique. A response that is grounded but answers the wrong question is still a failure.

---

## Value Object: RetrievedEvidence

### Consistency Invariants

- `context_recall_score` must be computed after every retrieval call, before evidence is injected into generation
- `context_precision_score` must be computed after every retrieval call, before evidence is injected into generation
- If `context_recall_score` falls below the minimum threshold, the retrieval query must be refined and retrieval retried — evidence with degraded recall must not proceed to generation
- Evidence must not be injected into generation if context quality scores indicate the retrieval failed to return the required knowledge

---

## Workflow Ordering (Cross-Cutting)

### Dependency Invariants

- Within every turn: intent must be classified **before** `DecisionContext` is updated
- Within every turn: `DecisionContext` must be updated **before** a `RetrievalQuery` is constructed
- Within every turn: context quality (`context_recall_score`, `context_precision_score`) must be evaluated **after** retrieval and **before** evidence injection
- Within every turn: groundedness and relevance scores must be evaluated **after** generation and **before** the response is returned
- Within every turn: retrieval must complete **before** response generation begins
- WF1 (Decision Context Building) must execute on every turn — it cannot be skipped or bypassed
