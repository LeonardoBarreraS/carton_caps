# Domain Concepts
## Carton Caps Conversational Assistant

---

## Extraction Source

Derived from: System Goal, Functional Requirements (FR1–FR8), System Workflows (WF0–WF3), and the provided database schema (Users, Schools, Products, Conversation_History, Purchase_History).

**Architecture constraint:** User authentication and registration are handled by an existing external REST API. This service receives every request with a pre-resolved, trusted `user_id`. The user's school is known at session start and pre-seeds the `DecisionContext` before any conversation turn.

---

## Process Entities

> Workflows that become domain objects with lifecycle and state transitions.

---

### ConversationSession

Represents the lifecycle of a multi-turn conversation between a user and the assistant.

It is created when the user sends the first message, evolves as turns accumulate, and ends when the session is closed. It holds the conversation history and the current decision context. All other workflows execute inside it.

**Lifecycle transitions:**
`created → initializing → idle ⇄ active → closed`

**Why a process entity:**
WF0 (Conversational Response Loop) is not a one-shot call. It is a long-running interaction process with state transitions. The session is the container that makes multi-turn conversation possible.

---

### DecisionContext

Represents the progressively built understanding of user preferences, constraints, and fundraising intent — accumulated across conversation turns.

It starts empty, grows with each user message, and reaches a state where it is rich enough to drive a precise retrieval query and a personalized recommendation.

**Lifecycle transitions:**
`empty → partial → ready → enriched`

**Key fields (domain aggregate state):**
- `school_anchor` — always present; never changes; pre-seeded at session init
- `_signals` — monotonically accumulated preference signals
- `_identified_gaps` — gaps detected during the last readiness evaluation
- `_pending_retrieval_intent` — the original retrieval intent saved when a clarification gap is detected (e.g. `"product_inquiry"`). Cleared once retrieval is successfully resumed after the user answers. Enables the clarification → resume loop across multiple turns.

**Why a process entity:**
WF1 (Decision Context Building) is a multi-step process with meaningful state transitions. The context is not just data — it is the core intelligence asset of the system that improves retrieval quality over time.

## Entities

> Objects with identity and lifecycle that persist across time.

---

### PurchaseHistory

The record of products previously purchased by the user, stored in the `Purchase_History` database table. Currently empty in the dataset. Designed as a **future context enrichment signal** — when populated, it can improve retrieval query construction by incorporating known purchase patterns (e.g., avoid recommending already-purchased items, suggest complementary products).

**Key attributes:** user_id, product_id, quantity, purchased_at

**Design decision:** The system is designed to query this at session initialization as an optional signal. If empty, it is skipped. If populated, it enriches the `DecisionContext`.

## Value Objects

> Immutable semantic values with no identity. Describe meaning, not things.

---

### User

**Type:** External Reference  
**Immutable.** A parent or shopper interacting with the assistant. Identity is pre-resolved by the external authentication API before any request reaches this service. The user's school association is loaded from the database at session initialization to pre-seed the `DecisionContext`. Represents an immutable reference to a User entity in external Auth BC — never created or mutated in this context.

**Key attributes:** identity, associated_school_id

**Note:** Authentication and registration are out of scope — handled by an existing external REST API.

---

### School

**Type:** External Reference  
**Immutable.** A school that the user supports through their purchases. Loaded from the database at session initialization. Provides the fundraising anchor for the `DecisionContext` before any conversation begins. Represents an immutable reference to a School entity in external Master Data BC — never created or mutated in this context.

**Key attributes:** identity, name, address

**Note:** The school is already known when the session starts — it does not need to be inferred from conversation.

---

### ConversationHistory

**Type:** Immutable Historical Entity  
**Immutable (append-only log).** The persisted record of all messages exchanged between the user and the assistant, stored in the `Conversation_History` database table. Loaded at session start to restore prior context when a user returns. Enables context continuity across sessions, not just within a single session. Created once, never mutated after creation.

**Key attributes:** user_id, message, sender (user / assistant), timestamp

---

### Intent

The classification of a user message into a recognized interaction type.

**Values:** `product_inquiry`, `referral_question`, `general_question`, `clarification_response`, `ambiguous`, `out_of_scope`

- **`general_question`** — on-domain questions about Carton Caps, the service, or the fundraising program (not specifically a product or referral mechanic).
- **`out_of_scope`** — messages clearly unrelated to Carton Caps: jokes, random chat, complaints, account/billing claims, or anything with no connection to shopping, products, referrals, or fundraising. Routes to a polite scope-redirect; bypasses the DI pipeline entirely.

---

### PreferenceSignal

A single preference or constraint extracted from one user message turn.

**Examples:** `kids=picky`, `preference=healthy`, `budget=sensitive`, `goal=support school`

---

### RetrievedEvidence

The content returned from a knowledge source after a retrieval query. Immutable — it is returned, not constructed. Carries context quality scores evaluated before injection into generation.

**Types:** product evidence, referral rule evidence

**Evaluation fields:**
- `context_recall_score` (float 0–1) — measures whether retrieval returned all relevant knowledge needed to answer the query. Computed after retrieval, before evidence injection.
- `context_precision_score` (float 0–1) — measures the proportion of retrieved chunks that are actually relevant. Low precision means noisy context enters the LLM.

**Design rule:** If `context_recall_score` falls below the minimum threshold, the retrieval query must be refined and retrieval retried before injection proceeds. Evidence with degraded context scores must not be passed to generation.

---

### AssistantResponse

The grounded text response returned to the user. Immutable once generated. Always derived from retrieved evidence and the current decision context. Carries evaluation scores that enforce grounding and relevance invariants.

**Evaluation fields:**
- `groundedness_score` (float 0–1) — measures whether every claim in the response is traceable to a retrieved evidence chunk. Replaces the boolean `grounded_in_evidence`. The boolean is derived: `groundedness_score >= threshold`.
- `relevance_score` (float 0–1) — measures whether the response actually addresses the user's intent. Computed using the reverse-question technique: "what question would this answer respond to?" compared semantically to the original intent.

**Design rule:** A response must not be returned to the caller if either `groundedness_score` or `relevance_score` falls below the minimum threshold. A safe fallback message is returned instead.

---

### Product

**Type:** External Reference Value Object  
**Immutable.** An item available in the platform's catalog. Retrieved by semantic relevance during the knowledge retrieval step. The assistant recommends only products returned by retrieval — it never invents or mutates products. Sourced from external knowledge base; never created or modified within this context.

**Key attributes:** identity (product_id), name, category, brand, description, attributes

**Invariant:** INV-AR-2 — product recommendations must cite only products returned by retrieval; never invented by the LLM.

---

### RetrievalQuery

**Type:** Immutable Historical Artifact  
**Immutable.** A context-enriched query constructed from the accumulated decision context and the detected intent. Never built directly from the raw user message. Derived, enriched, and purposeful — the direct input to the knowledge retrieval step. Created once via factory method; never mutated; represents a point-in-time context snapshot.

**Key attributes:** query_id, session_id, constructed query_text, source_target (product_catalog or referral_program_rules), originating_intent, context_state_at_creation, created_at

**Invariants:**
- INV-RQ-1 — query_text must be constructed from context, not the raw user message
- INV-RQ-2 — source_target derived from intent: product_inquiry → product_catalog; referral_question → referral_program_rules
- INV-RQ-3 — source_target must match originating_intent

---

### ReferralRule

A specific rule from the referral program (e.g., eligibility condition, bonus mechanics, invitation requirement). Immutable — defined by the platform, not the user.

## Domain Language

> The ubiquitous language of this system. These terms must be used consistently across all design and implementation artifacts.

| Term                  | Meaning                                                                                              |
|-----------------------|------------------------------------------------------------------------------------------------------|
| ConversationSession   | The stateful container of a multi-turn user interaction                                              |
| DecisionContext       | The accumulated structured understanding of user preferences and fundraising intent                  |
| PreferenceSignal      | A single piece of user intent or preference extracted from a message                                |
| Intent                | The classified purpose of a user message                                                             |
| RetrievalQuery        | A context-enriched query constructed to retrieve relevant evidence from a knowledge source           |
| RetrievedEvidence     | Content returned from a knowledge source; carries context_recall_score and context_precision_score before injection |
| Product               | A purchasable item from the platform catalog                                                         |
| School                | The school the user supports; known at session start; anchors the fundraising context                |
| ConversationHistory   | Persisted message records for a user, loaded at session start to restore prior context               |
| PurchaseHistory       | Persisted purchase records; optional context enrichment signal; empty now, designed for future use   |
| ReferralRule          | A platform-defined rule governing referral eligibility or rewards                                   |
| AssistantResponse     | The grounded, evidence-backed reply produced by the assistant; carries groundedness_score and relevance_score |
| User                  | A parent or shopper with pre-resolved identity, engaging the assistant to support their school       |
