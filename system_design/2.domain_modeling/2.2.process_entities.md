# Process Entity Identification
## Carton Caps Conversational Assistant

---

## Summary

Two process entities emerge from the workflows. Both were already identified in domain concept extraction. This document formally specifies their lifecycle, states, transitions, responsibilities, and domain events.

| Entity             | Type      | Derived From |
|--------------------|-----------|--------------|
| ConversationSession| Primary   | WF0          |
| DecisionContext    | Secondary | WF1          |

WF2 (Product Recommendation) and WF3 (Referral Guidance) are **not** independent process entities. They activate and complete within a single turn — they have no sustained lifecycle. They are behavioral steps inside `ConversationSession`.

---

## Primary Process Entity — ConversationSession

**Represents:**
The full lifecycle of a multi-turn interaction between a user and the assistant. It is the container that holds all state for one conversation — the current turn, the decision context, and the conversation history. All behavioral workflows execute inside it.

**Derived from:** WF0 — Conversational Response Loop

**Lifecycle:**
```
created → initializing → idle ⇄ active → closed
```

**States:**

| State          | Meaning                                                                            |
|----------------|------------------------------------------------------------------------------------|
| `created`      | Session record exists. User identity and session ID established.                   |
| `initializing` | User profile, school, ConversationHistory, and PurchaseHistory being loaded from DB. DecisionContext being pre-seeded. |
| `active`       | Session is processing a turn. A user message has been received and is being handled.|
| `idle`         | Turn completed. Session is waiting for the next user message.                      |
| `closed`       | Session has ended. No further turns will be processed.                             |

**State Transitions:**

```
created
  → initializing    [session init: load user profile, school, conversation history, purchase history]

initializing
  → idle            [initialization complete, DecisionContext pre-seeded]

idle
  → active          [new user message received]
  → closed          [session timeout or explicit close]

active
  → idle            [turn completed, response returned]
  → idle            [turn failed mid-execution — rollback via fail_turn(); no event emitted; user can retry]

closed
  [terminal state — no further transitions]
```

**Responsibilities:**
- Own the conversation lifecycle from creation to close
- Hold and provide access to the current `DecisionContext`
- Sequence conversation turns
- Persist turns to `ConversationHistory`
- Coordinate session initialization (loading DB data)

**Does NOT:**
- Execute retrieval
- Call LLM directly
- Route to workflows — that is application orchestration

**Domain Events:**

| Event                  | Emitted when                                              |
|------------------------|-----------------------------------------------------------|
| `SessionStarted`       | Session moves from `initializing` to `idle` (ready for first turn)|
| `TurnProcessed`        | Session moves from `active` to `idle` (turn completed)    |
| `SessionClosed`        | Session moves to `closed`                                 |

## Secondary Process Entity — DecisionContext

**Represents:**
The progressively built, structured understanding of the user's preferences, constraints, and fundraising intent. Starts pre-seeded with the user's school at session initialization. Grows with each conversation turn. Its state determines whether retrieval can be precise and whether a recommendation can be generated.

**Derived from:** WF1 — Decision Context Building

**Lifecycle:**
```
empty → partial → ready → enriched
```

**States:**

| State      | Meaning                                                                                          |
|------------|--------------------------------------------------------------------------------------------------|
| `empty`    | No signals yet. Only the school anchor is present (pre-seeded). Not sufficient for recommendation.|
| `partial`  | Some preference signals accumulated. Gaps still exist. Clarification may be needed.             |
| `ready`    | Sufficient signals to construct a meaningful retrieval query and generate a recommendation.      |
| `enriched` | Multiple turns have added depth. Retrieval queries are highly precise.                           |

**State Transitions:**

```
empty
  → partial     [first PreferenceSignal extracted from user message]

partial
  → partial     [more signals added, gaps remain]
  → ready       [gap evaluation passes — sufficient signals for recommendation]

ready
  → enriched    [further signals continue to improve retrieval precision]

enriched
  → enriched    [additional signals further refine context]
```

**Note:** Once `ready`, the context never regresses. Signals accumulate monotonically — they are never removed.

**What DecisionContext holds:**

| Field                 | Source                          | Always present? |
|-----------------------|---------------------------------|-----------------|
| `school`              | User profile (DB, session init) | Yes             |
| `preference_signals`  | Extracted from conversation     | After turn 1    |
| `fundraising_intent`  | Extracted from conversation     | When expressed  |
| `purchase_signals`    | PurchaseHistory (DB, session init)| Optional        |
| `identified_gaps`     | Gap evaluation step (WF1)       | When gaps exist |

**Responsibilities:**
- Accumulate `PreferenceSignal` values across turns
- Track identified gaps and determine when context is `ready`
- Expose its current state as the primary input for `RetrievalQuery` construction
- Anchor fundraising context via the pre-seeded school

**Does NOT:**
- Construct the retrieval query (that is an application-layer operation)
- Execute retrieval
- Generate responses

**Domain Events:**

| Event              | Emitted when                                                     |
|--------------------|------------------------------------------------------------------|
| `ContextUpdated`   | A new `PreferenceSignal` is added and state changes              |
| `GapDetected`      | Gap evaluation finds the context insufficient for recommendation |
| `ContextReady`     | Context transitions to `ready` state for the first time         |

---

## Relationship Between the Two Entities

`ConversationSession` **owns** `DecisionContext`.

- `ConversationSession` is created first
- During `initializing`, it creates and pre-seeds the `DecisionContext`
- Each turn in `ConversationSession` triggers WF1, which updates `DecisionContext`
- `DecisionContext.state` determines whether WF2 (Product Recommendation) activates

```
ConversationSession
    owns →  DecisionContext
                ↓ state drives
            RetrievalQuery construction
                ↓
            RAG pipeline (WF2 / WF3)
```
