# Context-to-Component Mapping
## Carton Caps Conversational Assistant

---

## Deployment Model

**Monolith — single deployable unit.**

All components run within the same process. Bounded context semantics are preserved through **module isolation**, not process isolation. No microservices, no queues between components, no separate deployments.

---

## Components

### COMP-1 — `conversation_management`

**Bounded Context:** BC-1 — Conversation Management  
**Module path:** `src/conversation_management/`  
**Orchestration engine:** LangGraph — `TurnGraph` executes the turn pipeline inside `ProcessTurnUseCase`. Nodes call ports only. Domain logic stays in the domain layer.

**Role:**  
Orchestrates the full conversation turn loop. Owns the session lifecycle from creation to close. Classifies intent on every turn. Loads user context at session start. Assembles and returns the grounded `AssistantResponse` to the API caller.

**Owns:**
| Kind | Domain Objects |
|------|----------------|
| Process Entities | `ConversationSession` |
| Entities | `ConversationHistory` (immutable historical entity — append-only log) |
| Value Objects | `User` (external reference), `School` (external reference), `Intent`, `AssistantResponse` |

**Enforces:** INV-CS-1 through INV-CS-7, INV-WF-1, INV-WF-3, INV-WF-4

**Calls:** `decision_intelligence` — sends classified `Intent` + current turn message on every turn  
**Receives from:** `decision_intelligence` — `RetrievedEvidence` for response injection

---

### COMP-2 — `decision_intelligence`

**Bounded Context:** BC-2 — Decision Intelligence  
**Module path:** `src/decision_intelligence/`  
**Orchestration engine:** LangGraph — `DecisionIntelligenceSubgraph` executes the decision pipeline inside `ProcessTurnIntelligenceUseCase`. Includes retrieval retry cycle on low recall and gap detection branch. Nodes call ports only.

**Role:**  
Accumulates preference signals monotonically across turns. Evaluates context readiness. Constructs an enriched `RetrievalQuery` from accumulated context — never from the raw message. Routes the query to the correct knowledge source. Passes resulting evidence back to Conversation Management.

**Owns:**
| Kind | Domain Objects |
|------|----------------|
| Process Entities | `DecisionContext` |
| Entities | `RetrievalQuery` |
| Value Objects | `PreferenceSignal`, `RetrievedEvidence` |

**Enforces:** INV-DC-1 through INV-DC-6, INV-RQ-1 through INV-RQ-3, INV-WF-2

**Calls:** `knowledge_retrieval` — sends context-enriched `RetrievalQuery` when context is partial/ready/enriched  
**Receives from:** `conversation_management` (Intent + message) and `knowledge_retrieval` (RetrievedEvidence as return value)

---

### COMP-3 — `knowledge_retrieval`

**Bounded Context:** BC-3 — Knowledge Retrieval  
**Module path:** `src/knowledge_retrieval/`

**Role:**  
Maintains the product catalog and referral program rules as semantically searchable knowledge sources. Executes retrieval for a given `RetrievalQuery`. Returns top-matching `RetrievedEvidence`. Does not invent or rewrite content — only retrieves.

**Owns:**
| Kind | Domain Objects |
|------|----------------|
| Entities | `Product`, `ReferralRule` |
| Value Objects | `RetrievedEvidence` |

**Enforces:** INV-AR-1 through INV-AR-4

**Calls:** nothing — leaf component with no outbound dependencies  
**Receives from:** `decision_intelligence` (RetrievalQuery)

---

### COMP-4 — `shell`

**Bounded Context:** none — infrastructure and wiring only  
**Module path:** `src/shell/`

**Role:**  
API entry points and composition root. Owns no domain logic. The only module that knows all components simultaneously. Bootstraps the application and wires dependencies.

**Owns:** REST API controllers, dependency injection / composition root, LangGraph checkpointer (SQLite-backed) — wired here and injected into `TurnGraph` and `DecisionIntelligenceSubgraph` to enable optional cross-session graph state persistence  
**Calls:** `conversation_management` — passes inbound API request (user_id + message)

---

## Dependency Graph

```
shell  ──►  conversation_management  ──►  decision_intelligence  ──►  knowledge_retrieval
```

- Dependencies flow strictly in one direction: orchestrator → executor
- Evidence flows **back as return values** — not as reverse dependencies
- `knowledge_retrieval` is the leaf: no outbound dependencies
- `shell` is the root: imports all components, owns no domain logic

---

## Folder Structure

```
src/
├── conversation_management/    ← BC-1: session lifecycle, intent, response assembly
├── decision_intelligence/      ← BC-2: context accumulation, query construction
├── knowledge_retrieval/        ← BC-3: semantic retrieval, product + referral rules
└── shell/                      ← entry points + wiring (no domain logic)
```

Each module maps **1:1** to a bounded context. `shell/` is the exception — it is the composition root, not a domain module.

---

## Boundary Rules

1. No component imports domain logic from another component's domain layer.
2. Cross-context value objects (`RetrievedEvidence`, `RetrievalQuery`) flow as **return values** through call chains — not as shared domain imports.
3. `User` and `School` are owned by `conversation_management`. `decision_intelligence` receives the school anchor as a pre-seeded value at session init — it does not import `School` from `conversation_management`.
4. `shell` is the only module allowed to import from multiple components simultaneously.
5. No circular dependencies permitted.

---

## RAG Pipeline as Component Flow

| Step | Component | Action |
|------|-----------|--------|
| 1 | `conversation_management` | Classifies intent on incoming message |
| 2 | `decision_intelligence` | Extracts signals, evaluates context, builds `RetrievalQuery` |
| 3 | `knowledge_retrieval` | Executes semantic retrieval, returns `RetrievedEvidence` |
| 4 | `decision_intelligence` | Receives evidence, passes it back |
| 5 | `conversation_management` | Injects evidence into generation, assembles `AssistantResponse` |

The RAG pipeline maps directly onto the component flow. Each stage is owned by exactly one component.
