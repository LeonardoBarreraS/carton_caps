# Part 1 — For Non-Technical Readers

## What We Built (Architecturally)

In this phase, we took everything we learned about the problem — the users, their needs, the fundraising goal, the conversation patterns — and translated it into a **concrete structure** for the software.

Think of it as designing the floor plan of a building before any brick is laid. This floor plan defines:

- How many distinct areas the system has
- What each area is responsible for
- How they talk to each other
- Which rules apply where
- How the pieces fit together into a working whole

The result is a **clean, organized system** where every part has a clear job — and nothing steps on someone else's territory.

---

## How the System Is Organized

The assistant is built as **one application** running as a single unit (not split across many services). Inside that single application, there are **four distinct components**, each with a clear and separate responsibility.

Imagine a well-run kitchen in a restaurant:

| Component | Kitchen Analogy | Real Role |
|-----------|----------------|-----------|
| **Conversation Management** | The head waiter | Greets the guest, manages the table, takes the order, delivers the meal, and closes the tab |
| **Decision Intelligence** | The sous-chef | Listens to what the guest wants, builds the recipe from accumulated knowledge, and sends the right order to the pantry |
| **Knowledge Retrieval** | The pantry | Holds and serves only real, verified ingredients — never improvises |
| **Shell** | The restaurant door + manager's office | The entrance for all guests (API) and the back office that organizes who does what |

Every guest interaction flows from the waiter → sous-chef → pantry and back. Each step builds on the previous one. Nothing is skipped.

---

## What Happens When a User Sends a Message

Here is what the system does, step by step, every single time a message arrives:

**1. The message is received**
The API entry point (the shell) passes the message to the Conversation Management component, which is responsible for the full conversation.

**2. The message is understood**
The system figures out what the user is asking: Are they looking for a product? Asking about referrals? Answering a clarifying question? This intent drives everything that follows.

**3. Preferences are noted**
Anything useful in the message — "picky kids," "tight budget," "prefer healthy" — is extracted and remembered. This builds a growing profile for this conversation.

**4. The system checks what it knows**
The Decision Intelligence component reviews everything accumulated so far. If there is enough to work with, it proceeds. If not, it prepares a focused clarifying question.

**5. A smart search is performed**
The system constructs a search query not from the raw message — but from everything it has learned about this user's needs. That enriched query is sent to the Knowledge Retrieval component.

**6. Real results are fetched**
The Knowledge Retrieval component searches the product catalog or the referral program rules and returns verified, real content. It never invents anything.

**7. The quality of the search is checked**
Before proceeding, the system verifies that what was retrieved is actually useful and relevant. If not, the search is refined and retried.

**8. A response is generated**
Using the retrieved content and the conversation context, an AI-generated response is produced.

**9. The response is checked before delivery**
Two quality checks run: Is everything in this response actually traceable to something retrieved (no hallucination)? Does it actually answer what the user asked? Only if both checks pass, the response is sent.

**10. The conversation continues**
The turn is recorded. The context grows. The next message will be even better understood.

---

## What Makes This Design Sound

**It has clear ownership.** Every piece of data — every session, every product, every user preference — belongs to exactly one component. Nothing is shared or duplicated.

**It has clear rules.** The architecture enforces business rules at the right level. For example: a response cannot be delivered unless it passes grounding and relevance checks. A conversation context can never lose signals it already accumulated. These are structural guarantees, not just hopes.

**It is safe to grow.** Because each component has strict boundaries, improving one part of the system does not risk breaking another. The retrieval logic can be upgraded without touching the conversation logic. The scoring logic can evolve without affecting the data model.

**It respects what exists.** The system never handles user authentication — that is done elsewhere by an existing service. The architecture explicitly acknowledges this boundary from the start.

**It is honest by design.** The Knowledge Retrieval component will never invent a product or a referral rule. What comes out of retrieval is what goes into the response — nothing more. The scoring gates enforce this structurally.

# Part 2 — Technical Summary

## Architecture Overview

| Property | Value |
|----------|-------|
| Architecture Pattern | Domain-Driven Design + Clean Architecture |
| Deployment Model | Monolith — single deployable unit |
| Internal Structure | Module isolation via bounded contexts |
| Layer Pattern | domain → application → infrastructure (inward dependencies only) |
| Cross-Context Communication | Synchronous direct interface calls via port contracts + DTO translators |
| Application Orchestration Engine | LangGraph — executes turn pipelines in COMP-1 and COMP-2 as explicit graphs with conditional edges and retry cycles |
| AI Integration | LLM adapters in infrastructure layer — behind port interfaces |
| Entry Point | REST API (shell/api) → COMP-1 application use cases |

## Component Structure

```
src/
├── conversation_management/    ← BC-1 | session lifecycle, intent, response assembly
│   ├── domain/
│   ├── application/
│   └── infrastructure/
├── decision_intelligence/      ← BC-2 | context accumulation, signal extraction, query construction
│   ├── domain/
│   ├── application/
│   └── infrastructure/
├── knowledge_retrieval/        ← BC-3 | semantic retrieval, product catalog, referral rules
│   ├── domain/
│   ├── application/
│   └── infrastructure/
└── shell/
    ├── api/                    ← REST entry points, DTOs, controllers
    └── composition/            ← Composition root — wires all components; no domain logic
        └── adapters/           ← Cross-context adapters (only place with multi-component imports)
```

## Component Roles and Ownership

### COMP-1 — `conversation_management` (BC-1)

**Role:** Master orchestrator of the conversation turn loop.

| Layer | Key Artifacts |
|-------|---------------|
| Domain | `ConversationSession` (process entity — state machine: created → initializing → idle ⇄ active → closed), `ConversationHistory`, `User`, `School`, `Intent` (VO), `AssistantResponse` (VO) |
| Application | `StartSessionUseCase`, `ProcessTurnUseCase` (delegates to `TurnGraph`), `CloseSessionUseCase` + `TurnGraph` (LangGraph — classifies intent, routes conditionally, generates and evaluates response) + `TurnGraphState` + ports: `IIntentClassifier`, `IResponseGenerator`, `IResponseEvaluator`, `IDecisionIntelligenceService` |
| Infrastructure | DB adapters (session, history, user, school, purchase history), `LLMIntentClassifier`, `LLMResponseGenerator`, `LLMResponseEvaluator` |

**Invariants enforced:** INV-CS-1 through INV-CS-7 (session state machine, turn sequencing, one active session per user), INV-AR-1 / INV-AR-5 / INV-AR-6 (response must pass groundedness and relevance score thresholds before delivery), INV-WF-1, INV-WF-3, INV-WF-4.

---

### COMP-2 — `decision_intelligence` (BC-2)

**Role:** Progressive preference accumulation and context-enriched retrieval query construction.

| Layer | Key Artifacts |
|-------|---------------|
| Domain | `DecisionContext` (process entity — state machine: empty → partial → ready → enriched), `RetrievalQuery` (entity), `PreferenceSignal` (VO), `RetrievedEvidence` (VO) |
| Application | `PreSeedContextUseCase`, `ProcessTurnIntelligenceUseCase` (delegates to `DecisionIntelligenceSubgraph`) + `DecisionIntelligenceSubgraph` (LangGraph — extracts signals, evaluates context, retrieves, retries on low recall, detects gaps) + `DecisionGraphState` + ports: `ISignalExtractor`, `IEvidenceEvaluator`, `IKnowledgeRetrievalService` |
| Infrastructure | `InMemoryDecisionContextRepository` (session-scoped), `LLMSignalExtractor`, `LLMEvidenceEvaluator` |

**Invariants enforced:** INV-DC-1 through INV-DC-6 (context never regresses, school always present, signals never removed, monotonic accumulation), INV-RQ-1 through INV-RQ-3 (query always constructed from context, source_target must match intent routing), INV-WF-2.

---

### COMP-3 — `knowledge_retrieval` (BC-3)

**Role:** Semantic retrieval leaf. Executes vector store searches. Returns real content only.

| Layer | Key Artifacts |
|-------|---------------|
| Domain | `Product` (entity), `ReferralRule` (entity), `RetrievedEvidence` (VO), `IProductRepository`, `IReferralRuleRepository` |
| Application | `ExecuteRetrievalUseCase` — routes by `source_target` to the correct repository |
| Infrastructure | `ProductVectorStoreRepository`, `ReferralRuleVectorStoreRepository` |

**Invariants enforced:** INV-AR-1 through INV-AR-4 (retrieved content only — never invented, never rewritten by LLM).

No outbound dependencies. Leaf node in the dependency graph.

---

### COMP-4 — `shell`

**Role:** System entry point and composition root. Owns zero domain logic.

| Sub-layer | Artifacts |
|-----------|-----------|
| `api/` | `ConversationController`, `MessageRequestDTO`, `AssistantResponseDTO` |
| `composition/` | `Container` (dependency injection / wiring), `DecisionIntelligenceAdapter`, `KnowledgeRetrievalAdapter` |

The `Container` is the only artifact in the system that imports from multiple components simultaneously.

## Dependency Graph

```
COMP-4 (shell)
    ↓
COMP-1 (conversation_management)
    ↓  [IDecisionIntelligenceService port]
COMP-2 (decision_intelligence)
    ↓  [IKnowledgeRetrievalService port]
COMP-3 (knowledge_retrieval)
```

Strictly unidirectional. Evidence flows back as return values through the call chain — not as reverse dependencies. No bidirectional imports. No circular references.

---

## Cross-Context Communication

**Pattern: Synchronous direct interface calls via owned port contracts + DTO translators.**

Rationale: the RAG pipeline is inherently synchronous — each step requires the result of the previous one. Event-driven async communication would add complexity without benefit.

| Channel | Port | From | To | Adapter location |
|---------|------|------|----|-----------------|
| COMM-1 | `IDecisionIntelligenceService` | COMP-1 application | COMP-2 application | `shell/composition/adapters/` |
| COMM-2 | `IKnowledgeRetrievalService` | COMP-2 application | COMP-3 application | `shell/composition/adapters/` |

**Port ownership rule:** The caller owns and defines the port interface. The implementation lives in the callee, wired via adapter in the shell.

**DTO translators (5 total):**

| DTO | Direction | Purpose |
|-----|-----------|---------|
| `PreSeedContextDTO` | COMP-1 → COMP-2 | School anchor + purchase signals at session init |
| `TurnInputDTO` | COMP-1 → COMP-2 | Intent (as string) + message per turn |
| `TurnIntelligenceResultDTO` | COMP-2 → COMP-1 | Evidence chunks, quality scores, context state, clarification signal |
| `RetrievalQueryDTO` | COMP-2 → COMP-3 | Constructed query text + source_target routing |
| `RetrievedEvidenceDTO` | COMP-3 → COMP-2 | Raw retrieved chunks + evidence type |

No domain object from any component's domain layer ever crosses a context boundary.

## RAG Pipeline Mapped to Architecture

| Pipeline Step | Component | Layer | Key Artifact |
|--------------|-----------|-------|--------------|
| Receive request | COMP-4 | api | `ConversationController` |
| Classify intent | COMP-1 | infra → application | `LLMIntentClassifier` via `IIntentClassifier` |
| Extract preference signals | COMP-2 | infra → application | `LLMSignalExtractor` via `ISignalExtractor` |
| Accumulate context | COMP-2 | domain | `DecisionContext` state machine |
| Construct retrieval query | COMP-2 | application | `ProcessTurnIntelligenceUseCase` → `RetrievalQuery` |
| Execute semantic retrieval | COMP-3 | infra → application | `QdrantProductRepository` / `QdrantReferralRuleRepository` |
| Evaluate evidence quality | COMP-2 | infra → application | `LLMEvidenceEvaluator` via `IEvidenceEvaluator` (recall + precision gates) |
| Generate grounded response | COMP-1 | infra → application | `LLMResponseGenerator` via `IResponseGenerator` |
| Evaluate response quality | COMP-1 | infra → application | `LLMResponseEvaluator` via `IResponseEvaluator` (groundedness + relevance gates) |
| Return validated response | COMP-1 → COMP-4 | application → api | `AssistantResponse` → `AssistantResponseDTO` |

## Quality Gate Architecture

Two evaluation stages are structurally enforced — neither is optional:

**Stage 1 — Evidence Quality (COMP-2, after retrieval)**
- `context_recall_score`: were all relevant chunks retrieved?
- `context_precision_score`: are all retrieved chunks relevant?
- If recall < threshold → query is refined and retrieval retried before proceeding

**Stage 2 — Response Quality (COMP-1, after generation)**
- `groundedness_score`: is every claim in the response traceable to retrieved evidence?
- `relevance_score`: does the response actually answer the user's intent? (reverse-question technique)
- If either < threshold → response is blocked; safe fallback returned to caller

These gates are not application-level checks — they are enforced by domain invariants (INV-AR-1, INV-AR-5, INV-AR-6, INV-RE-1, INV-RE-2).

## Key Architectural Decisions

| Decision | Rationale |
|----------|-----------|
| **Monolith over microservices** | Single deployment unit eliminates distributed system complexity (network, serialization, distributed tracing) for a system where all components are tightly coupled by the synchronous RAG pipeline |
| **Module isolation, not service isolation** | Semantic boundaries preserved through code organization — not deployment topology |
| **Port interfaces owned by the caller** | Each component defines what it needs, not how the other works. Enables independent evolution of implementations without changing contracts |
| **Adapters exclusively in shell/composition** | The single place in the codebase with cross-component visibility. All other layers are strictly scoped to their own component |
| **DTO translators at every boundary** | No domain object from one component's domain layer ever enters another component's domain layer. Semantic leakage is structurally impossible |
| **DecisionContext as in-memory repository** | Context accumulation is session-scoped by design — decision context lives as long as the session and requires no cross-restart persistence at prototype scale. The design allows adding a persistent repository later without changing the domain model. |
| **LLM behind port interfaces** | AI services are infrastructure detail. Domain and application layers never know which model is used — swap models without changing business logic |
| **Domain events intra-component only** | the synchronous RAG pipeline has no use case for async cross-context events. GapDetected surfaces as a DTO field, not a published event |
| **LangGraph as application-layer execution engine** | The turn pipeline in COMP-1 and the decision intelligence pipeline in COMP-2 are inherently graph-shaped: conditional routing by intent, retrieval retry cycles, gap detection branches. LangGraph is adopted strictly at the application layer — nodes are thin port-callers, domain logic is untouched, port interfaces are unchanged. The architecture already described a graph; LangGraph runs it. |
