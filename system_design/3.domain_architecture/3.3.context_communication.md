# Context Communication Design
## Carton Caps Conversational Assistant

---

## Communication Model

**Synchronous Direct Interface Calls via Ports.**

The RAG pipeline requires evidence before generating a response — the flow is inherently synchronous and each step depends on the previous result. Async event-driven cross-context communication would add complexity with no benefit here.

All cross-context calls:
- are synchronous (caller blocks for result)
- cross via **port interfaces** defined in the caller's application layer
- use **plain DTOs** as translators — no domain objects cross boundaries
- are **bridged by adapters** that live exclusively in `src/shell/composition/adapters/`

Domain events are **intra-component signals only** — they do not drive cross-context communication.

---

## Cross-Context Communication Map

```
shell/api
    │
    ▼
COMP-1: conversation_management
    │  IDecisionIntelligenceService (port)
    │  DecisionIntelligenceAdapter (shell wiring)
    ▼
COMP-2: decision_intelligence
    │  IKnowledgeRetrievalService (port)
    │  KnowledgeRetrievalAdapter (shell wiring)
    ▼
COMP-3: knowledge_retrieval
```

All adapters live in `src/shell/composition/adapters/` — the one place in the system permitted to import across component boundaries.

## COMM-1 — Conversation Management → Decision Intelligence

**Port:** `IDecisionIntelligenceService`  
**Defined in:** `src/conversation_management/application/ports/`  
**Adapter:** `DecisionIntelligenceAdapter` in `src/shell/composition/adapters/`  
**Triggers:** WF0 (every turn), session initialization

COMP-1 owns the contract. It defines what it needs from COMP-2 — COMP-2's internal mechanics (DecisionContext, PreferenceSignal) are invisible to COMP-1.

### Methods

| Method | Called by | When |
|--------|-----------|------|
| `preSeedContext(PreSeedContextDTO)` | `StartSessionUseCase` | Once per session, during initialization, before first turn |
| `processTurn(TurnInputDTO) → TurnIntelligenceResultDTO` | `ProcessTurnUseCase` | Every turn, after intent classification |

---

### Transfer DTOs — COMM-1

**`PreSeedContextDTO`** *(COMP-1 → COMP-2, defined in COMP-1 application/ports)*

| Field | Type | Source |
|-------|------|--------|
| `session_id` | string | Active ConversationSession identity |
| `school_id` | string | School value object loaded at session init |
| `school_name` | string | School value object loaded at session init |
| `purchase_signals` | list[string] | PurchaseHistory (empty list if no records) |

> School value object (external reference) never crosses the boundary. Only its minimal data is passed.

---

**`TurnInputDTO`** *(COMP-1 → COMP-2, defined in COMP-1 application/ports)*

| Field | Type | Source |
|-------|------|--------|
| `session_id` | string | Active ConversationSession |
| `intent` | string | Intent value (string representation — not the Intent object) |
| `message` | string | Raw user message text |
| `turn_number` | int | Current turn index within the session |
| `conversation_history` | list[dict] | Full conversation history loaded from `IConversationHistoryRepository.find_by_user_id()`, mapped to OpenAI messages format `[{"role": "user"|"assistant", "content": str}]`. Empty list if this is the first turn. Passed to BC-2 so all LLM calls receive full conversational context (Proposal B — explicit history injection). |

---

**`TurnIntelligenceResultDTO`** *(COMP-2 → COMP-1, defined in COMP-1 application/ports)*

| Field | Type | Source |
|-------|------|--------|
| `evidence_chunks` | list[string] | RetrievedEvidence content from COMP-3 |
| `evidence_type` | string | `product_evidence` or `referral_rule_evidence` |
| `context_recall_score` | float | Evaluated by COMP-2 `IEvidenceEvaluator` |
| `context_precision_score` | float | Evaluated by COMP-2 `IEvidenceEvaluator` |
| `context_state` | string | DecisionContext current state: `empty`\|`partial`\|`ready`\|`enriched` |
| `clarification_needed` | boolean | Whether COMP-2 detected a context gap |
| `clarification_question` | string \| null | Suggested question if `clarification_needed` is true |

> No DecisionContext, PreferenceSignal, or RetrievalQuery objects cross into COMP-1. `GapDetected` (a COMP-2 domain event) surfaces here as the `clarification_needed` field — it never travels as a cross-context event.

## COMM-2 — Decision Intelligence → Knowledge Retrieval

**Port:** `IKnowledgeRetrievalService`  
**Defined in:** `src/decision_intelligence/application/ports/`  
**Adapter:** `KnowledgeRetrievalAdapter` in `src/shell/composition/adapters/`  
**Triggers:** WF2 (product_inquiry intent), WF3 (referral_question intent)

COMP-2 owns the contract. It defines what retrieval it needs — COMP-3's internal mechanics (Product, ReferralRule, vector store) are invisible to COMP-2.

### Methods

| Method | Called by | When |
|--------|-----------|------|
| `retrieve(RetrievalQueryDTO) → RetrievedEvidenceDTO` | `ProcessTurnIntelligenceUseCase` | When context is partial/ready/enriched and intent warrants retrieval |

---

### Transfer DTOs — COMM-2

**`RetrievalQueryDTO`** *(COMP-2 → COMP-3, defined in COMP-2 application/ports)*

| Field | Type | Source |
|-------|------|--------|
| `query_text` | string | `RetrievalQuery.constructed_text` |
| `source_target` | string | `RetrievalQuery.source_target` — `product_catalog` or `referral_program_rules` |

> The `RetrievalQuery` entity (COMP-2 domain) is never passed to COMP-3. Only its data is translated into this DTO.

---

**`RetrievedEvidenceDTO`** *(COMP-3 → COMP-2, defined in COMP-2 application/ports)*

| Field | Type | Source |
|-------|------|--------|
| `chunks` | list[string] | Top-matching content from vector store |
| `evidence_type` | string | `product_evidence` or `referral_rule_evidence` |

> COMP-3 returns raw retrieved content. COMP-2 maps this into its own `RetrievedEvidence` value object and runs quality evaluation (recall + precision scores) before passing upstream.

## Adapter Layer — `src/shell/composition/adapters/`

Both cross-context adapters live here. This is the only place in the codebase that imports from multiple components simultaneously.

| Adapter | Implements | Delegates to | Translation responsibility |
|---------|-----------|-------------|---------------------------|
| `DecisionIntelligenceAdapter` | `IDecisionIntelligenceService` (COMP-1) | `PreSeedContextUseCase`, `ProcessTurnIntelligenceUseCase` (COMP-2) | Maps COMP-1 DTOs → COMP-2 use case inputs; COMP-2 outputs → COMP-1 DTOs |
| `KnowledgeRetrievalAdapter` | `IKnowledgeRetrievalService` (COMP-2) | `ExecuteRetrievalUseCase` (COMP-3) | Maps COMP-2 DTOs → COMP-3 use case input; COMP-3 output → COMP-2 DTO |

The `Container` (composition root) instantiates both adapters at startup and injects them into their respective use cases via constructor injection.

## Domain Events — Intra-Component Only

Domain events in this system do not drive cross-context communication. They are lifecycle signals within each component's own domain — useful for observability and local state transitions. The shell layer drains them after each use-case boundary and writes them to `data/domain_events.log` through `LoggingEventPublisher`.

| Component | Event | Scope | Notes |
|-----------|-------|-------|-------|
| COMP-1 | `SessionStarted` | Intra-component | Emitted when session enters active state; logged by shell `LoggingEventPublisher` |
| COMP-1 | `TurnProcessed` | Intra-component | Emitted when turn completes; logged by shell `LoggingEventPublisher` |
| COMP-1 | `SessionClosed` | Intra-component | Emitted when session closes; logged by shell `LoggingEventPublisher` |
| COMP-2 | `ContextUpdated` | Intra-component | Emitted when a PreferenceSignal is added; logged by shell `LoggingEventPublisher` |
| COMP-2 | `GapDetected` | Intra-component | Logged by shell `LoggingEventPublisher` and surfaced to COMP-1 as `clarification_needed` field in DTO — never as a cross-context event |
| COMP-2 | `ContextReady` | Intra-component | Emitted when context first reaches `ready` state; logged by shell `LoggingEventPublisher` |

## Per-Turn Communication Sequence

```
shell/api  ──►  ProcessTurnUseCase (COMP-1)
                    │
                    │  classify intent (IIntentClassifier — COMP-1 infra)
                    │
                    │  IDecisionIntelligenceService.processTurn(TurnInputDTO)
                    ▼
            DecisionIntelligenceAdapter (shell/composition/adapters)
                    │
                    ▼
            ProcessTurnIntelligenceUseCase (COMP-2)
                    │
                    │  extract signals → update DecisionContext → build RetrievalQuery
                    │
                    │  IKnowledgeRetrievalService.retrieve(RetrievalQueryDTO)
                    ▼
            KnowledgeRetrievalAdapter (shell/composition/adapters)
                    │
                    ▼
            ExecuteRetrievalUseCase (COMP-3)
                    │  vector store retrieval
                    ▼
            RetrievedEvidenceDTO ──►  KnowledgeRetrievalAdapter ──►  COMP-2
                    │
                    │  evaluate evidence quality (IEvidenceEvaluator — COMP-2 infra)
                    ▼
            TurnIntelligenceResultDTO ──►  DecisionIntelligenceAdapter ──►  COMP-1
                    │
                    │  generate response (IResponseGenerator)
                    │  evaluate response (IResponseEvaluator)
                    ▼
            AssistantResponse ──►  shell/api  ──►  HTTP response
```

## Session Initialization Sequence

```
shell/api  ──►  StartSessionUseCase (COMP-1)
                    │
                    │  load User, School, ConversationHistory, PurchaseHistory
                    │
                    │  IDecisionIntelligenceService.preSeedContext(PreSeedContextDTO)
                    ▼
            DecisionIntelligenceAdapter (shell/composition/adapters)
                    │
                    ▼
            PreSeedContextUseCase (COMP-2)
                    │
                    │  create DecisionContext
                    │  set school anchor + purchase signals
                    ▼
            Session transitions to active — ready for first turn
```

## Boundary Rules

1. No domain object from any component's domain layer crosses a context boundary.
2. Every cross-context payload is a plain DTO defined adjacent to the port interface it serves.
3. Port interfaces are owned and defined by the **caller** (in caller's application/ports layer).
4. Adapters that bridge contexts live exclusively in `src/shell/composition/adapters/`.
5. No component's domain or application layer imports from another component's domain layer.
6. `GapDetected` is a domain signal internal to COMP-2 — it surfaces as a DTO field, never as a cross-context event.
