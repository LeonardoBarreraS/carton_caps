# Requirements Summary
## Carton Caps Conversational Assistant

---

## What Did We Build in This Phase?

Phase 0 established *what the problem is* and *who the system serves*. Phase 1 turns that understanding into a **precise contract** ‚Äî a definition of exactly what the system must do and how it must behave.

This phase produced two artifacts that every subsequent phase depends on:

1. **Functional Requirements (FR1‚ÄìFR8)** ‚Äî Eight capabilities the system must provide, each with observable behaviors and clear rationale
2. **System Workflows (WF0‚ÄìWF3)** ‚Äî Four behavioral workflows that define how the assistant's capabilities activate and combine at runtime

These are not implementation specifications. They do not describe databases, APIs, or AI models. They describe *behavior* ‚Äî what a user experiences, and what guarantees the system makes.

---

## Why Does This Phase Matter?

Requirements are the contract between the problem and the solution. Without them, teams build things that work technically but don't solve the right problem.

For this system, requirements matter especially because the assistant could be built in many ways. A simple chatbot would satisfy some requirements. A search-over-products tool would satisfy others. The requirements here define **what combination of behaviors** creates a system that genuinely influences fundraising outcomes ‚Äî and that combination is specific, non-trivial, and not obvious without careful analysis.

# Part 1 ‚Äî For Non-Technical Readers

## What must the assistant do?

The assistant has eight core capabilities:

---

### 1. Remember conversations (Session Management)

The assistant does not start fresh with every message. When you open a conversation, it already knows:
- Who you are
- Which school you support
- What you've said in previous conversations

It holds this context for the entire conversation ‚Äî and the context grows with every message you send.

---

### 2. Understand what you're asking (Intent Recognition)

Before responding to anything, the assistant figures out what kind of message you sent:

| What you said | What the assistant understands |
|---|---|
| "What should I buy for my kids?" | You're looking for a product recommendation |
| "How does the referral program work?" | You want to know about inviting others |
| "What is Carton Caps anyway?" | You have a general question |
| "Healthy options" (after it asked a question) | You're answering its clarifying question |
| "Tell me a joke" | You're off-topic; the assistant will redirect gently |

Getting this right is what allows the assistant to route your message to the right behavior instead of guessing.

---

### 3. Learn what you need (Decision Context Construction)

This is the core of how the assistant gets smarter over time. Every message you send contains signals ‚Äî clues about what you need, what your constraints are, and what your family is like.

**Turn by turn:**

| Your message | What the assistant learns |
|---|---|
| "My kids are picky" | `kids = picky` |
| "Also something healthy" | `preference = healthy` |
| "What should I buy?" | Enough context to recommend specific products |

These signals accumulate. They never disappear once learned. Each new message makes the next recommendation more precise.

---

### 4. Search for answers, not make them up (Knowledge Retrieval)

When the assistant recommends products or explains the referral program, it is not inventing answers from its AI training. It is actively searching:

- When you ask about products ‚Üí it searches the product catalog
- When you ask about referrals ‚Üí it searches the referral program rules

And it does not search with your raw words. It builds a *smarter* search query from everything it has learned about you. If it knows you want healthy snacks for picky kids, it searches for that ‚Äî not just for "snacks."

After searching, it checks the quality of what it found before using it. If the results are not good enough, it refines the search and tries again. Only verified, relevant results reach your answer.

---

### 5. Recommend products that are right for you (Product Recommendation)

Product recommendations must meet three requirements:

- They must come from the actual product catalog (no invented products)
- They must match what the assistant has learned about your preferences
- Every recommendation must explicitly mention that your purchase supports your school

*Example:* "These healthy snack options are a great fit for picky kids ‚Äî and every purchase supports Lincoln Elementary."

---

### 6. Explain how referrals work (Referral Guidance)

When you ask about inviting others or helping your school more, the assistant explains the referral program using the platform's actual rules ‚Äî not a generic explanation. This means the guidance is always accurate and specific to how Carton Caps actually works.

---

### 7. Ask one good question when it needs more information (Clarifying Questions)

If you ask for a recommendation but the assistant does not yet know enough to give a good one, it asks one focused question ‚Äî not a survey.

*Example:* If you say "I want to buy something" with no context, the assistant asks: *"Are you looking for snacks, beverages, or household products?"* ‚Äî one question, directly targeting the most important gap.

---

### 8. Respond through an API (Conversational Endpoint)

The assistant is a backend service, not a standalone app. It exposes a clean API that the mobile application communicates with. The API accepts a user message and session identifier, and returns the assistant's response along with quality information.

---

## How does a conversation actually flow?

Every single message you send goes through this sequence, in order, every time:

```
You send a message
    ‚Üì
Assistant classifies your intent
    ‚Üì
Assistant extracts new signals and updates your decision context
    ‚Üì
If the assistant needs more information ‚Üí asks one clarifying question
    ‚Üì
If you asked about products or referrals ‚Üí searches the right knowledge source
    ‚Üì
Checks the quality of what was found (and retries if needed)
    ‚Üì
Generates a response grounded in the retrieved evidence
    ‚Üì
Checks that the response is accurate and actually answers your question
    ‚Üì
Response delivered
```

No step is optional. No step is skipped. The two quality checks ‚Äî on the *search results* and on the *generated response* ‚Äî are what prevent the assistant from making things up or giving irrelevant answers.

# Part 2 ‚Äî Technical Summary

## Functional Requirements ‚Äî Complete Definition

### FR1 ‚Äî Conversation Session Management

The system must initiate and maintain a conversation session that preserves context across multiple user turns.

**Session initialization (before the first user message):**
- Load authenticated user's profile from the database
- Resolve the user's associated school
- Load prior `ConversationHistory` (if any)
- Load `PurchaseHistory` signals (if populated)
- Pre-seed `DecisionContext` with school and history

**Architecture assumption:** User identity is pre-resolved by the existing external REST API. Every request arrives with a trusted `user_id`. This service never authenticates.

**Invariants:**
- One active session per user at any time
- Session lifecycle: `created ‚Üí initializing ‚Üí idle ‚áÑ active ‚Üí closed`
- `closed` is terminal ‚Äî no recovery
- `active` ‚Üí `closed` is illegal; session must return to `idle` first

---

### FR2 ‚Äî Intent Recognition

The system must classify the intent of every incoming user message before any processing begins.

**Intent taxonomy:**

| Intent | Example | System Behavior |
|---|---|---|
| `product_inquiry` | "What should I buy for my kids?" | Activates WF2 (product retrieval) |
| `referral_question` | "How does the referral program work?" | Activates WF3 (referral retrieval) |
| `general_question` | "What is Carton Caps?" | Handled inline; no retrieval |
| `clarification_response` | User answering assistant's question | Signals fed into context update |
| `ambiguous` | Unclear or mixed signals | One clarifying question returned |
| `out_of_scope` | "Tell me a joke", weather, billing claims | Polite redirect; pipeline bypassed entirely |

**Out-of-scope handling:** Messages in this category skip the Decision Intelligence pipeline and retrieval entirely. A polite scope-redirect response is returned. This prevents off-domain messages from entering the RAG pipeline and producing confusing fallback responses.

---

### FR3 ‚Äî Progressive Decision Context Construction

The system must extract preference signals from each user message and accumulate them into a structured decision context that persists across all turns of the session.

**Signal types extracted:**

| Signal Category | Examples |
|---|---|
| Product preferences | `category=snacks`, `preference=healthy`, `budget=low` |
| Family context | `kids=picky`, `dietary_restriction=no_nuts` |
| Fundraising goals | `goal=maximize_school_impact`, `referral_interest=high` |
| Meal context | `meal_occasion=breakfast`, `health_goal=balanced` |

**Pre-seeded at session start (before any user turn):**
- School name and identity ‚Äî from user's DB profile
- Prior conversation signals ‚Äî from `ConversationHistory` if prior sessions exist
- Purchase signals ‚Äî from `PurchaseHistory` if populated (graceful degradation when empty)

**Context lifecycle:** `empty ‚Üí partial ‚Üí ready ‚Üí enriched`
- Signals accumulate monotonically ‚Äî never removed once added
- Context never regresses: once `ready`, it cannot return to `partial`
- `ready` means: sufficient signals exist to produce a meaningful retrieval query

**Relationship to retrieval:** The accumulated decision context is the direct input to retrieval query construction. A richer context ‚Üí more precise query ‚Üí more relevant evidence ‚Üí better response.

---

### FR4 ‚Äî Context-Enriched Knowledge Retrieval

The system must construct a retrieval query from the current intent and accumulated decision context, retrieve from the correct knowledge source, evaluate quality, and inject passing evidence into response generation.

**Full retrieval pipeline:**

| Step | Description |
|---|---|
| Query construction | Retrieval query built from `intent + DecisionContext` ‚Äî never from the raw user message alone |
| Source routing | `product_inquiry` ‚Üí product catalog; `referral_question` ‚Üí referral program rules |
| Semantic retrieval | Vector similarity search returns the most relevant content |
| Context quality evaluation | `context_recall_score` (coverage) and `context_precision_score` (relevance) computed on `RetrievedEvidence` |
| Retry loop | If `recall < threshold` and retries remain, query is refined and retrieval retried (up to `max_retries=2`; `top_k` escalated per retry) |
| Evidence injection | Retrieved content (with passing quality scores) passed to response generation |
| Response quality evaluation | `groundedness_score` (claim traceability) and `relevance_score` (intent alignment) computed on `AssistantResponse` |
| Quality gate | If either score falls below threshold, response is blocked; safe fallback returned instead |

**Knowledge sources:**
- **Product catalog** ‚Äî queried on `product_inquiry` intent or when context is `ready`
- **Referral program rules** ‚Äî queried on `referral_question` intent or when fundraising opportunity detected

**Quality thresholds:**
- Minimum groundedness: `0.8`
- Minimum relevance: `0.75`
- Minimum context recall: `0.6` (triggers retrieval retry if not met)

---

### FR5 ‚Äî Context-Driven Product Recommendation

The system must generate personalized product recommendations grounded exclusively in retrieved product evidence, and explicitly connect every recommendation to the user's school fundraising goal.

**Requirements:**
- Recommendations must cite only retrieved products ‚Äî no LLM-invented product names
- Every recommendation must reference the user's school by name
- `groundedness_score` must be verified before delivery
- `relevance_score` must be verified before delivery

---

### FR6 ‚Äî Referral Guidance

The system must provide accurate referral program guidance grounded in retrieved referral rules ‚Äî not LLM background knowledge.

**Requirements:**
- Referral rules are retrieved from the knowledge base on every referral question
- Guidance covers mechanics, eligibility, and suggested next action
- LLM memory about referral programs is never used as the source

---

### FR7 ‚Äî Clarifying Question Generation

The system must detect context gaps and respond with one focused clarifying question when insufficient context prevents a good recommendation.

**Gap detection rule (for `product_inquiry`):**
A gap is flagged only when the context contains *none* of the five retrieval-useful signal keys: `product_category`, `meal_occasion`, `dietary_restriction`, `health_goal`, `budget_tier`. Any single key is sufficient to proceed to retrieval. If none are present, one open-ended clarifying question is asked.

**For `referral_question`:** Context readiness check is bypassed entirely ‚Äî referral questions always proceed directly to retrieval regardless of context state.

**Behavioral rule:** One question at a time. Never a questionnaire.

---

### FR8 ‚Äî Conversational API Endpoint

The system must expose a REST API for multi-turn conversation management.

**Endpoints:**

| Method | Path | Purpose |
|---|---|---|
| `POST` | `/sessions` | Open a new conversation session; returns session identifier |
| `POST` | `/sessions/{id}/turns` | Send a message; receive grounded assistant response with quality scores |
| `DELETE` | `/sessions/{id}` | Close the session cleanly |

**Response envelope for `POST /sessions/{id}/turns`:**
- `response_text` ‚Äî the assistant's grounded answer
- `intent` ‚Äî classified intent for this turn
- `is_fallback` ‚Äî whether the quality gate failed and a safe fallback was returned
- `groundedness_score` ‚Äî claim traceability score
- `relevance_score` ‚Äî intent alignment score

## System Workflows

### Workflow Overview

| ID | Name | Type | Trigger |
|---|---|---|---|
| WF0 | Conversational Response Loop | Primary | Every user message |
| WF1 | Decision Context Building | Secondary | Every user message |
| WF2 | Product Recommendation | Secondary | `product_inquiry` intent OR context is `ready` |
| WF3 | Referral Guidance | Secondary | `referral_question` intent OR fundraising opportunity detected |

**Session initialization (before WF0's first turn):**
Runs once when a session opens. Loads user, school, `ConversationHistory`, `PurchaseHistory`. Pre-seeds `DecisionContext`. This ensures the first turn already has full user context.

---

### WF0 ‚Äî Conversational Response Loop

> Primary workflow. Executes on every user message. Orchestrates all secondary workflows.

```
receive message
    ‚Üí classify intent
    ‚Üí update context              [WF1 always activates]
    ‚Üí construct retrieval query   [intent + accumulated context ‚Üí enriched query]
    ‚Üí retrieve evidence           [WF2 or WF3 based on intent and context state]
    ‚Üí evaluate context quality    [recall + precision gate on RetrievedEvidence]
    ‚Üí generate grounded response
    ‚Üí evaluate response quality   [groundedness + relevance gate on AssistantResponse]
    ‚Üí return validated response
```

| Step | Contract |
|---|---|
| `classify intent` | Every message is classified before any other processing |
| `construct retrieval query` | Query is always `intent + context` ‚Äî never the raw user message |
| `evaluate context quality` | Must pass before evidence is injected into generation |
| `evaluate response quality` | Must pass before response is returned to caller |

---

### WF1 ‚Äî Decision Context Building

> Secondary workflow. Mandatory on every turn.

```
message received
    ‚Üí signals extracted
    ‚Üí context updated
    ‚Üí gaps evaluated
    ‚Üí [if gaps and product intent] one clarifying question returned
    ‚Üí context ready
```

Pre-condition: `DecisionContext` already seeded with school and prior history at session start.

**Context state transitions triggered here:**
- `empty ‚Üí partial` (first non-school signal added)
- `partial ‚Üí partial` (further signals)
- `partial ‚Üí ready` (gap evaluation passes for the first time)
- `ready ‚Üí enriched` (signals continue accumulating after readiness)

---

### WF2 ‚Äî Product Recommendation

> Secondary workflow. Activates when product intent is detected or context is ready.

```
context evaluated
    ‚Üí retrieval query constructed   [from intent + accumulated context]
    ‚Üí product catalog searched      [semantic vector search]
    ‚Üí context quality evaluated     [recall + precision; retry if recall < threshold]
    ‚Üí evidence injected             [only if quality passes]
    ‚Üí recommendation generated      [grounded; cites only retrieved products]
    ‚Üí school connection added
    ‚Üí response returned
```

**Example:**
```
Context: product_category=snacks, health_goal=healthy, school=Lincoln Elementary
‚Üí retrieval query: "healthy snack options for families"
‚Üí product catalog searched; top results retrieved
‚Üí recall score: 0.82 (passes); precision score: 0.79 (passes)
‚Üí evidence injected
‚Üí "These healthy snack options are great for families ‚Äî every purchase supports Lincoln Elementary."
```

---

### WF3 ‚Äî Referral Guidance

> Secondary workflow. Activates on referral intent or fundraising opportunity.

```
referral intent detected
    ‚Üí retrieval query constructed   [from intent + fundraising context]
    ‚Üí referral rules retrieved      [platform-specific rules from knowledge base]
    ‚Üí context quality evaluated     [recall + precision; retry if recall < threshold]
    ‚Üí evidence injected             [only if quality passes]
    ‚Üí guidance formulated           [strictly from retrieved rules]
    ‚Üí specific referral action suggested
    ‚Üí response returned
```

**Bypass rule:** For `referral_question` intent, context readiness is not checked ‚Äî retrieval proceeds even when `DecisionContext` is `empty`. This prevents forcing a clarification round when a user asks a factual referral question on the first message.

## Key Design Decisions Established Here

**1. Context enrichment is the defining architectural insight.**
The retrieval query is never the raw user message ‚Äî it is always constructed from the accumulated `DecisionContext`. This single decision is what makes retrieval quality improve with each turn and what separates this system from a stateless chatbot.

**2. Two quality gates, not one.**
Evidence quality (recall + precision) is evaluated *before* generation. Response quality (groundedness + relevance) is evaluated *after* generation. Both must pass before a response reaches the user. These gates are not optional checks ‚Äî they are structural invariants enforced at the application layer.

**3. Out-of-scope handling is a first-class requirement.**
Off-domain messages do not enter the retrieval pipeline. This protects response quality and maintains the system's focused purpose.

**4. Referral questions bypass context readiness.**
A user should be able to ask "how does the referral program work?" as their first message and receive a grounded, specific answer ‚Äî not a clarifying question. The bypass rule ensures this works correctly.

**5. Clarification is one question, not a questionnaire.**
The system asks the most impactful single question when a gap exists. It never interrogates the user with a list of questions.

**6. The API is behavior-transparent.**
Every response from `POST /sessions/{id}/turns` includes the quality scores for that turn ‚Äî groundedness, relevance, and whether a fallback was returned. This supports monitoring, debugging, and evaluation without requiring additional tooling.
