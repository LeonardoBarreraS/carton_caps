# System Workflows
## Carton Caps Conversational Assistant

---

## Design Principle

The conversation never resets. Context accumulates across all turns. Behavioral workflows activate based on detected intent and context state — all inside the same continuous session.

At the core of every response is a RAG loop: the accumulated decision context is used to construct a precise retrieval query, which retrieves grounded evidence, which is injected into the response generation step. Context makes retrieval smarter. Retrieval makes generation accurate.

---

## RAG Core Pattern (applied in WF2 and WF3)

```
decision context + intent
    → construct retrieval query
    → retrieve relevant evidence
    → evaluate context quality     [recall + precision scores]
    → inject evidence into generation
    → generate grounded response
    → evaluate response quality    [groundedness + relevance scores]
    → return validated response
```

The retrieval query is never just the raw user message.
It is always enriched by the accumulated decision context.
Evaluation gates enforce that retrieval quality and response quality meet thresholds before proceeding.

---

## Workflow Overview

| ID  | Name                           | Type      | Trigger                                         |
|-----|--------------------------------|-----------|-------------------------------------------------|
| WF0 | Conversational Response Loop   | Primary   | Every user message                              |
| WF1 | Decision Context Building      | Secondary | Every user message                              |
| WF2 | Product Recommendation         | Secondary | Product intent OR context is ready              |
| WF3 | Referral Guidance              | Secondary | Referral intent OR fundraising opportunity      |

**Note:** Before WF0 begins its first turn in a session, a session initialization step loads the user's profile, school, conversation history, and purchase history (if any) from the database. This pre-seeds `DecisionContext` before the first user message is processed.

## WF0 — Conversational Response Loop

> **Primary workflow.** Executed on every user message. Orchestrates all behavioral workflows.

```
receive message
    → classify intent
    → update context              [activates WF1]
    → construct retrieval query   [context + intent → enriched query]
    → retrieve evidence           [activates WF2 or WF3 based on intent]
    → evaluate context quality    [recall + precision check on RetrievedEvidence]
    → generate grounded response
    → evaluate response quality   [groundedness + relevance check on AssistantResponse]
    → return response
```

**Entry:** User sends a message with a session identifier
**Exit:** Assistant response returned to the user
**Loop:** Repeats for every conversation turn

| Step                          | Description                                                                 |
|-------------------------------|-----------------------------------------------------------------------------|
| receive message               | User message arrives with session identifier                                |
| classify intent               | System determines the type of interaction (FR2)                             |
| update context                | Signals extracted and decision context enriched (FR3)                       |
| construct retrieval query     | Query built from intent + accumulated context — not raw message (FR4)      |
| retrieve evidence             | Relevant content fetched from the appropriate knowledge source (FR4)        |
| evaluate context quality      | `context_recall_score` and `context_precision_score` computed on `RetrievedEvidence`. If recall < threshold, query is refined and retrieval retried. |
| generate grounded response    | Response produced using retrieved evidence + decision context               |
| evaluate response quality     | `groundedness_score` and `relevance_score` computed on `AssistantResponse`. If either < threshold, response is blocked and a safe fallback returned. |
| return response               | Validated response and context snapshot delivered to caller                 |

## WF1 — Decision Context Building

> **Secondary workflow.** Runs inside every turn. Builds the progressive understanding of the user.

```
message received
    → signals extracted
    → context updated
    → gaps evaluated
    → [if gaps exist] clarification requested
    → context ready
```

**Trigger:** Any user message
**Purpose:** Progressively understand user needs, preferences, and fundraising intent

**Pre-condition:** At session start, `DecisionContext` is already seeded with:
- the user's school (from DB)
- prior conversation signals (from `ConversationHistory` if any)
- purchase signals (from `PurchaseHistory` if populated)

WF1 then enriches this context further with each new user turn.

| Transition               | Meaning                                                        |
|--------------------------|----------------------------------------------------------------|
| signals extracted        | Preferences and constraints identified from the message        |
| context updated          | Decision context enriched with new signals                     |
| gaps evaluated           | System checks whether context is sufficient for a recommendation |
| clarification requested  | One targeted question asked when gaps exist (optional step)    |
| context ready            | Sufficient context available to activate WF2                   |

**Example:**
```
User: "My kids are picky"
→ signal extracted: kids=picky
→ context updated
→ gap: product type unknown
→ clarification: "Are you looking for snacks or breakfast items?"
```

## WF2 — Product Recommendation

> **Secondary workflow.** Activates when product intent is detected or decision context is sufficiently built.

```
context evaluated
    → retrieval query constructed   [from intent + decision context]
    → products retrieved            [catalog queried with enriched query]
    → context quality evaluated     [recall + precision scores on RetrievedEvidence]
    → evidence injected             [only if quality scores pass threshold]
    → recommendation generated      [grounded in retrieved evidence]
    → response formulated
```

**Trigger:** `product_inquiry` intent OR decision context is ready (WF1 exited with `context ready`)
**Purpose:** Guide the user toward a personalized product decision grounded in retrieved product data

| Transition                    | Meaning                                                                                    |
|-------------------------------|--------------------------------------------------------------------------------------------|
| context evaluated             | Decision context checked for sufficient preference signals                                 |
| retrieval query constructed   | Query built from context (e.g., "healthy snacks for picky kids"), not raw message          |
| products retrieved            | Product catalog queried with enriched query; top results returned                          |
| context quality evaluated     | `context_recall_score` and `context_precision_score` computed. If recall < threshold, query is refined and retrieval retried. |
| evidence injected             | Retrieved products (with passing quality scores) passed as grounded context into generation|
| recommendation generated      | Personalized recommendation produced citing only retrieved products                        |
| response formulated           | Response structured with product suggestions and preference-aligned reasoning              |

**Example:**
```
Context: kids=picky, preference=healthy
→ retrieval query: "healthy snacks suitable for picky children"
→ product catalog returns top matching items
→ evidence injected into generation
→ "These healthy snack options work well for picky kids: [retrieved products]…"
```

## WF3 — Referral Guidance

> **Secondary workflow.** Activates when the user expresses interest in referrals, helping their school, or when a fundraising opportunity is detected.

```
referral intent detected
    → retrieval query constructed   [from intent + fundraising context]
    → referral rules retrieved       [program rules fetched from knowledge base]
    → context quality evaluated      [recall + precision scores on RetrievedEvidence]
    → evidence injected              [only if quality scores pass threshold]
    → guidance formulated            [grounded in retrieved rules]
    → action suggested
    → response returned
```

**Trigger:** `referral_question` intent OR fundraising support opportunity detected in conversation
**Purpose:** Explain referral mechanics and encourage action using retrieved, platform-specific rules

| Transition                    | Meaning                                                                |
|-------------------------------|------------------------------------------------------------------------|
| referral intent detected      | User signals interest in helping their school or in referrals          |
| retrieval query constructed   | Query built from intent and fundraising context signals                |
| referral rules retrieved      | Platform-specific referral rules fetched from knowledge base           |
| context quality evaluated     | `context_recall_score` and `context_precision_score` computed. If recall < threshold, query is refined and retrieval retried. |
| evidence injected             | Retrieved rules (with passing quality scores) passed as grounded context into generation |
| guidance formulated           | Explanation built strictly from retrieved rules, not LLM memory        |
| action suggested              | Specific referral action recommended based on retrieved eligibility    |
| response returned             | Guidance and suggested action delivered to the user                    |

**Example:**
```
User: "How do I help my school more?"
→ referral intent detected
→ retrieval query: "referral program rules school donation bonus"
→ referral rules retrieved
→ evidence injected into generation
→ "Inviting other parents increases school donations. Here is how the referral program works: [retrieved rules]…"
```

## Workflow Interaction Model

All secondary workflows run inside the primary loop. A single user message can activate WF1 plus one additional workflow depending on intent.

```
User Message
     |
     v
WF0: Conversational Response Loop
     |
     +---> WF1: Decision Context Building      [always activated]
     |          |
     |          +--> [if context ready + product intent] --> WF2: Product Recommendation
     |
     +---> [if referral intent] --> WF3: Referral Guidance
     |
     v
Assistant Response
```

**Routing rules:**

| Condition                                        | Workflow Activated |
|--------------------------------------------------|-------------------|
| Always                                           | WF1               |
| `product_inquiry` intent OR context is ready     | WF2               |
| `referral_question` or fundraising intent        | WF3               |

**Note:** WF2 and WF3 can both activate in the same turn if the user message contains both product and referral signals.
