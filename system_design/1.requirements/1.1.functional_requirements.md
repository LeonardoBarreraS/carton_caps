# Functional Requirements
## Carton Caps Conversational Assistant

---

## System Purpose

A stateful conversational fundraising co-pilot that learns user preferences and intent through dialogue and guides users toward product purchases and referral actions that increase school donations.

It is not a chatbot, a search interface, or a Q&A system. It is a decision-guidance system that uses domain data and conversation context to actively influence fundraising behavior.

---

## Actors

| Actor                | Role                                                                                        |
|----------------------|---------------------------------------------------------------------------------------------|
| User                 | Authenticated parent / shopper ‚Äî identity pre-resolved by the external REST API             |
| External Auth API    | Existing REST API handling authentication and registration. Out of scope for this service.  |
| System               | Conversational assistant that processes and responds                                        |
| Knowledge Base       | Product catalog and referral rules available as data source                                 |
| Database             | Stores users, schools, products, conversation history, and purchase history                 |

---

## FR1 ‚Äî Conversation Session Management

**The system must** initiate and maintain a conversation session that preserves context across multiple user turns.

**Session initialization:**
When a session starts, the system loads the authenticated user's profile from the database, resolves their associated school, and loads their prior `ConversationHistory`. This pre-seeds the `DecisionContext` with the school before any conversation turn begins.

**Architecture assumption:**
User identity is pre-resolved by the existing external REST API. Every request arrives with a trusted `user_id`. This service does not handle authentication or registration.

**Observable behavior:**
Each user message is processed within a continuous session. Prior messages influence subsequent responses. The session persists until explicitly ended. A returning user's prior conversation context is restored from the database.

**Rationale:**
Without session continuity, each turn is isolated and context cannot accumulate. Pre-seeding with school and history means the first assistant response is already personalized.

## FR2 ‚Äî Intent Recognition

**The system must** classify the intent of each incoming user message before processing.

**Recognized intent types:**

| Intent                  | Example trigger                                                      |
|-------------------------|----------------------------------------------------------------------|
| `product_inquiry`       | "What should I buy for my kids?"                                     |
| `referral_question`     | "How does the referral program work?"                                |
| `general_question`      | "What is Carton Caps?", "How does this service work?"                |
| `clarification_response`| User answering a question the assistant asked                        |
| `ambiguous`             | Unclear or mixed signals across on-domain categories                 |
| `out_of_scope`          | "Tell me a joke", "What's the weather?", "This app is terrible"      |

**Out-of-scope handling:**
Messages classified as `out_of_scope` (jokes, random chat, complaints, account or billing claims, insults, or any message with no connection to grocery shopping, products, referrals, or fundraising) receive a polite scope-redirect response. The DI pipeline and retrieval are bypassed entirely.

**Observable behavior:**
A user message is classified first. The classification determines which system behavior is activated next. Out-of-scope messages are handled gracefully without surfacing a confusing fallback.

**Rationale:**
Routing conversation to the correct capability requires knowing what the user wants. Adding `out_of_scope` prevents off-domain messages from entering the retrieval pipeline and generating poor or fallback responses.

## FR3 ‚Äî Progressive Decision Context Construction

**The system must** extract preference signals from each user message and accumulate them into a structured decision context across the conversation.

**Decision context captures:**

- Product preferences (e.g., healthy, snacks, budget-sensitive)
- Family context (e.g., picky kids, allergies, age groups)
- Fundraising goals (e.g., support school, maximize impact)
- Expressed constraints (e.g., avoid certain ingredients, price range)

**Pre-seeded at session start (before any user turn):**
- School name and identity ‚Äî loaded from the user's profile (no need to ask)
- Prior conversation signals ‚Äî loaded from `ConversationHistory` if the user has prior sessions
- Purchase signals ‚Äî loaded from `PurchaseHistory` if populated (currently empty; optional enrichment)

**Observable behavior:**
Turn 1 ‚Äî User: "My kids are picky" ‚Üí context: `kids=picky`
Turn 2 ‚Äî User: "Also healthy" ‚Üí context: `kids=picky, preference=healthy`
Turn 3 ‚Äî User: "What should I buy?" ‚Üí assistant uses full accumulated context.

**Relationship to retrieval:**
The accumulated decision context is the direct input to retrieval query construction. A richer context produces a more precise retrieval query, which produces more relevant evidence, which produces a better response. This is the core feedback loop of the system.

**Rationale:**
This transforms the assistant from a stateless chatbot into a decision co-pilot that learns the user across the conversation. It also makes every retrieval step smarter as the conversation progresses.

## FR4 ‚Äî Context-Enriched Knowledge Retrieval

**The system must** construct a retrieval query from the current intent and accumulated decision context, retrieve the most relevant knowledge from the appropriate source, and inject the retrieved evidence into the response generation step.

**Retrieval pipeline:**

| Step                          | Description                                                                 |
|-------------------------------|-----------------------------------------------------------------------------|
| Query construction            | A retrieval query is built from intent + decision context, not just the raw user message |
| Source routing                | Intent determines which knowledge source to query (products or referral rules) |
| Retrieval                     | The most semantically relevant content is retrieved from the selected source |
| Context quality evaluation    | `context_recall_score` and `context_precision_score` computed on `RetrievedEvidence`. If recall < threshold, query is refined and retrieval retried before injection. |
| Evidence injection            | Retrieved content (with passing quality scores) is passed to the response generation step as grounded context |
| Response quality evaluation   | `groundedness_score` and `relevance_score` computed on `AssistantResponse`. If either < threshold, response is blocked and a safe fallback message is returned. |

**Knowledge sources:**

- Product catalog ‚Äî queried when intent is `product_inquiry` or context is ready
- Referral program rules ‚Äî queried when intent is `referral_question` or fundraising opportunity detected

**Observable behavior:**
Context: `kids=picky, preference=healthy`
‚Üí Retrieval query: "healthy snacks suitable for picky children" (not just the raw user message)
‚Üí Product catalog queried with enriched query
‚Üí Top matching products retrieved; `context_recall_score` and `context_precision_score` evaluated
‚Üí If context quality passes, products injected before generation
‚Üí `groundedness_score` and `relevance_score` evaluated on the generated response
‚Üí Response returned only if both scores meet thresholds; otherwise a fallback message is returned
‚Üí Response is grounded in retrieved evidence, not LLM memory

**Rationale:**
Using the accumulated decision context to construct the retrieval query ‚Äî rather than the raw user message alone ‚Äî significantly improves retrieval precision and response relevance. This is the key RAG insight: better context input ‚Üí better retrieval ‚Üí better generation.

## FR5 ‚Äî Context-Driven Product Recommendation

**The system must** generate personalized product recommendations grounded exclusively in retrieved product evidence and the accumulated decision context, and connect each recommendation explicitly to the user's school fundraising goal.

**Observable behavior:**
Context: `kids=picky, preference=healthy, school=Lincoln Elementary`
‚Üí Enriched retrieval query constructed and products retrieved (FR4)
‚Üí Context quality evaluated: recall and precision scores computed before injection
‚Üí Assistant: "These healthy snack options are a great fit for picky kids ‚Äî and every purchase supports Lincoln Elementary."
‚Üí Cites only retrieved products. Reasoning ties to user's stated preferences AND their school.
‚Üí `groundedness_score` verified: response traces to retrieved evidence only, not LLM memory
‚Üí `relevance_score` verified: response addresses the user's actual intent, not a related but different question

**Rationale:**
Recommendations must be traceable to retrieved evidence. The fundraising connection must be explicit in every recommendation ‚Äî this is the co-pilot's core business purpose, not a detail.

---

## FR6 ‚Äî Referral Guidance

**The system must** provide contextual guidance about the referral program and its fundraising impact when referral intent is detected, grounded in retrieved referral rules.

**Observable behavior:**
User: "How can I help my school more?"
‚Üí Referral intent detected
‚Üí Referral program rules retrieved from knowledge base (FR4)
‚Üí Assistant explains mechanics, eligibility, and next action using retrieved content ‚Äî not generic LLM knowledge.

**Rationale:**
Referral rules are specific to the platform. Grounding this response in retrieved rules prevents incorrect guidance and builds user trust.

---

## FR7 ‚Äî Clarifying Question Generation

**The system must** detect when the decision context is insufficient to generate a good recommendation, and respond with one focused clarifying question.

**Observable behavior:**
User: "I want to buy something."
‚Üí Context is empty
‚Üí Assistant: "Are you looking for snacks, beverages, or household products?"
‚Üí One question at a time, not a questionnaire.

**Rationale:**
Asking targeted questions fills context gaps efficiently without overwhelming the user.

---

## FR8 ‚Äî Conversational API Endpoint

**The system must** expose the multi-turn conversation through a clean API endpoint that accepts a user message and session identifier, and returns the assistant response with the updated context state.

**Observable behavior:**
`POST /chat` with `{ session_id, message }` ‚Üí returns `{ response, context_snapshot }`

**Rationale:**
The API is the integration surface for the mobile application. It must be stateful and conversation-aware.
